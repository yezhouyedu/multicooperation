# 线上部署学习与实操路线

更新时间：2026-05-30

## 1. 先回答现在最重要的三个问题

## 问题 1：本地项目还没写完全，现在能不能谈部署？

可以谈，但要分清楚两件事：

- 现在可以开始搭服务器、学 Linux、学 Docker、走备案
- 但现在还不能假装项目已经具备“一键上云”的条件

结合当前仓库现状，项目离真正可复用的线上部署，还差至少这些东西：

1. 前端 `web` 还没有明确的线上 `Dockerfile`
2. 后端 `server` 还没有明确的线上 `Dockerfile`
3. 仓库里目前只有 PostgreSQL 的 Compose 文件
4. 线上环境变量方案还没定型
5. 域名、HTTPS、反向代理还没接
6. 数据持久化、备份、升级回滚流程还没定

### 当前仓库里已经有的东西

- 本地数据库容器编排：`infra/docker/docker-compose.yml`
- 本地后端环境变量模板：`apps/server/.env.example`
- 本地启动脚本：`scripts/start-local.ps1`

### 这意味着什么

当前项目更像：

- PostgreSQL 用 Docker 起
- 前端和后端仍主要按本地开发方式跑

也就是说，现在并不是“整个项目已经容器化，只差上传服务器”。

## 问题 2：我 Linux 和线上知识太缺，怎么恶补，而且不能盲信 AI？

这个判断非常对。

AI Code 最危险的不是“不会写命令”，而是：

- 你不知道命令会改什么
- 你分不清临时修复和长期方案
- 你分不清开发环境、测试环境、生产环境

所以你现在最应该建立的不是“背很多命令”，而是一个判断框架：

### 你每次看到 AI 给命令时，都问这 5 个问题

1. 这条命令是在改“当前目录文件”还是“系统配置”？
2. 这条命令会不会影响登录、网络、防火墙、磁盘、数据库？
3. 这条命令是临时生效还是永久生效？
4. 这条命令执行后，我应该用什么命令验证？
5. 如果出问题，怎么回退？

只要你开始这样问，AI 就更像副驾驶，不是方向盘。

### 现在最值得恶补的知识顺序

先后顺序不要反。

1. Linux 基础命令和目录结构
2. SSH 登录和文件传输
3. Docker 基础概念
4. Docker Compose
5. 环境变量和 `.env`
6. Nginx / 反向代理 / 域名 / HTTPS
7. GitHub Actions 或部署面板

不要一上来就学 Kubernetes。

## 问题 3：后续线上部署时，数据库和变量是不是会不一样？有没有一键打包方法？

会不一样，而且这是最正常的地方。

### 线上和本地通常至少有这几类差异

1. 数据库地址不一样
2. API 域名不一样
3. 环境变量值不一样
4. 文件路径不一样
5. 日志和持久化方式不一样
6. HTTPS 和端口暴露方式不一样

例如你现在后端模板里有：

- `DATABASE_URL`
- `SERVER_PORT`
- `OPENAI_BASE_URL`
- `OPENAI_API_KEY`
- `OPENAI_MODEL`

这些值本地和线上几乎一定不同。

### 有没有“一键打包到线上”的方法？

有，但有前提。

真正的一键部署，一般至少要先具备下面其中一种：

#### 方案 A：Dockerfile + docker compose

仓库里要有：

- 前端 Dockerfile
- 后端 Dockerfile
- 线上 compose 文件
- `.env.production` 或等价变量方案

这样以后在服务器上可以：

```bash
docker compose up -d
```

#### 方案 B：用自托管部署面板

比如：

- Coolify
- Dokploy
- CapRover

它们能帮你做：

- Git 拉代码
- 自动构建
- 填环境变量
- 管理域名和 HTTPS
- 重启和更新应用

但注意：

它们也不是魔法。
如果你的项目连最基本的构建入口都没有，它们也救不了。

### 对当前项目的现实判断

当前最靠谱的未来路线是：

1. 先把前端和后端的 Dockerfile 补出来
2. 把数据库、后端、前端的 compose 方案定下来
3. 再决定是直接 Docker Compose 上云
4. 还是接 Coolify / Dokploy 这类面板

## 2. 我对当前项目的部署判断

基于仓库现状，我更建议你把部署分成三层理解：

### 第一层：服务器基础设施

你现在已经在做，而且做得对：

- 买国内服务器
- 买域名
- 备案
- 登录 Ubuntu
- 装 Docker

这层是任何路线都绕不开的。

### 第二层：项目可部署化

这是接下来最关键的一层，目前还没做完：

- 前后端镜像怎么构建
- 生产模式怎么启动
- 环境变量如何组织
- 数据库如何连
- 哪些目录要持久化

### 第三层：部署自动化

等第二层定下来后，再考虑：

- GitHub Actions 自动部署
- Coolify / Dokploy 面板化部署
- 回滚、备份和升级

不要把第三层当成第一层。

## 3. 给新手的学习路线

下面这条路线是为“我不想盲信 AI，但也不想从零读一整本 Linux 教科书”的人设计的。

## 第 1 阶段：先建立最低限度理解

目标：

- 你知道自己在服务器上做了什么
- 你看得懂 70% 的基础命令

建议资料：

1. Ubuntu 官方新手命令行教程  
   链接：[The Linux command line for beginners](https://documentation.ubuntu.com/desktop/en/latest/tutorial/the-linux-command-line-for-beginners/)

2. Ubuntu 官方 CLI 速查表  
   链接：[Ubuntu CLI cheat sheet PDF](https://assets.ubuntu.com/v1/d00791ae-ubuntu_cli_cheat_sheet_2025.pdf)

3. Ubuntu Server 官方文档总入口  
   链接：[Ubuntu Server docs](https://ubuntu.com/server/docs/reference/)

这一阶段你至少要知道这些命令是干什么的：

- `pwd`
- `ls`
- `cd`
- `cat`
- `cp`
- `mv`
- `rm`
- `mkdir`
- `sudo`
- `apt update`
- `apt install`
- `systemctl status`
- `journalctl`
- `ps`
- `top`
- `df -h`
- `free -h`

## 第 2 阶段：学 SSH，不然你永远像在蒙眼开车

目标：

- 知道密码登录和密钥登录区别
- 知道 `.pem` 是什么
- 知道怎么从本地连远程服务器

建议资料：

1. 阮一峰《SSH 入门教程》  
   链接：[《SSH 入门教程》发布了](https://www.ruanyifeng.com/blog/2020/12/ssh-tutorial.html)

这一阶段你至少要理解：

- 公钥 / 私钥
- `.pem` 私钥不能乱发
- `ssh`
- `scp`
- `sftp`

## 第 3 阶段：学 Docker，不然你只是在照抄命令

目标：

- 分清镜像、容器、卷、网络
- 看懂 `docker run`、`docker ps`、`docker logs`
- 理解为什么“本地能跑”和“服务器能跑”不是一回事

建议资料：

1. Docker 官方 Get Started  
   链接：[Docker Get Started](https://www.docker.com/gettingstarted/)

2. Docker 官方安装文档（Ubuntu）  
   链接：[Install Docker Engine on Ubuntu](https://docs.docker.com/installation/ubuntulinux/)

3. Docker Compose 官方文档  
   链接：[Docker Compose docs](https://docs.docker.com/compose/)

4. Docker Compose Quickstart  
   链接：[Compose Quickstart](https://docs.docker.com/compose/gettingstarted/)

5. 中文 GitHub 教材：《Docker — 从入门到实践》  
   链接：[GitHub 项目索引](https://github.com/yeasy/docker_practice)  
   在线 PDF 索引镜像之一：[Docker 从入门到实践 PDF](https://yujunhao8831.github.io/images/docker_practice.pdf)

这一阶段你至少要能解释：

- 镜像是什么
- 容器是什么
- 卷是什么
- `docker compose up -d` 做了什么
- 为什么要挂载数据卷保存数据库

## 第 4 阶段：学“项目如何上线”，不是学“服务器如何存在”

目标：

- 知道部署是把“代码、配置、数据库、域名、证书”串起来

建议重点学：

1. 环境变量
2. 反向代理
3. 域名解析
4. HTTPS
5. 生产环境日志查看

建议资料：

1. GitHub Actions 官方 Quickstart  
   链接：[Quickstart for GitHub Actions](https://docs.github.com/en/actions/get-started/quickstart)

2. Docker 官方 GitHub Actions 文档  
   链接：[Docker Build GitHub Actions](https://docs.docker.com/build/ci/github-actions/)

### 方案 3：CapRover

适合：

- 想要偏经典、生态成熟一点的自托管 PaaS

官方与代码：

### 现在应该立刻上这些吗？

不建议立刻上。

原因：

- 你现在先要理解服务器和 Docker
- 当前项目也还没完成完整容器化
- 太早上面板，容易变成“只是把不会的东西藏起来”

我对你当前阶段的建议是：

1. 先用手工命令部署一遍最小版本
2. 自己看懂每一步在干什么
3. 然后再考虑接 Coolify / Dokploy

## 5. 给你的一条“最不容易翻车”的实战路线

### 第一步：把服务器基础搞定

你已经做了很多：

- 服务器已买
- Ubuntu 已确认
- Docker 已安装成功

接下来继续补：

- 学会本地 SSH 连接
- 学会查看服务状态
- 学会看磁盘、内存、日志

### 第二步：把项目整理到“可部署”

目标不是马上上线，而是让项目至少具备：

- `web/Dockerfile`
- `server/Dockerfile`
- 一个线上 compose 文件
- 明确的 `.env.production`

### 第三步：先手工部署一次

哪怕这次部署很笨，也值得。

因为你会真正知道：

- 哪些变量变了
- 数据库怎么接
- 镜像怎么 build
- 端口怎么暴露
- 哪一步最容易错

### 第四步：再自动化

等你手工跑通以后，再决定：

- GitHub Actions
- Coolify
- Dokploy

## 6. 你现在最需要掌握的，不是“更多命令”，而是“验证习惯”

以后每做一步，都要配一个验证动作。

例如：

- 装 Docker 后：`docker --version`
- 起服务后：`docker ps`
- 改配置后：`cat 配置文件`
- 重启服务后：`systemctl status`
- 配域名后：`ping` 或浏览器访问
- 配 HTTPS 后：浏览器实际打开

你只要坚持“改一步，验一步”，就会比很多只会抄教程的人稳得多。

## 7. 当前一句话建议

你现在最适合的路线不是“立刻找一个一键部署神器”，而是：

- 先把 Linux / SSH / Docker 的底层理解补到能自保
- 再把这个项目补到可部署
- 最后再决定要不要用 Coolify / Dokploy 做自动化

如果只给一个最现实的执行建议：

1. 先看 Ubuntu 新手命令行教程
2. 再看 SSH 入门教程
3. 再看 Docker Get Started 和 Docker Compose Quickstart
4. 同时把当前项目整理到“前后端各有 Dockerfile”
5. 然后我再陪你做第一次手工上线
