## 开篇：先把话说清楚

氛围编程（Vibe Coding）本身没问题，**问题往往在使用它的人**。

很多人听说可以跟 AI 编程助手聊天就交付软件，于是以为自己掌握了魔法：

- 打开 AI
- 用一句话描述想法
- 期待它自动生成一个完整、稳定、漂亮的产品

结果通常是：

- 一堆屎山代码
- 幻觉出来的 UI 和配色
- 页面路由不通
- 看似“能跑”但一碰就碎的 App

然后你开始怪 AI。

但真相是：

> **AI 产生幻觉，不是因为它坏了，而是因为你没给它任何可以依靠的结构。**

没有结构。没有清晰度。没有基础。

Vibe Coding 是有效的，**但前提是你必须理解自己在构建什么，并给 AI 一个完整的工作系统。**

这篇文章讲的不是零碎技巧，而是一整套系统：

- 构建模块
- 基础词汇
- 文档系统
- 工作流程
- 工具选择
- 发布前验证
- 长期维护方法

如果你看完还是做不出来，那多半不是信息不够，而是执行力度不够。

# 一、为什么你失败了

你失败，是因为你**跳过了基础**。

你可能并不真正理解这些概念：

- 什么是组件
- 什么是状态（state）
- 为什么按钮点击没反应
- 为什么电脑上看着正常，手机上一塌糊涂

正因为你自己没搞清楚，你也就没法准确告诉 AI 你要什么。

## AI 的本质：翻译器，而不是魔法师

AI 的工作方式是：

> **把你的意图翻译成代码。**

如果你的意图本身是模糊的、混乱的、残缺的，那生成出来的代码也一定会是混乱的。

如果你无法清晰表达需求，AI 就只能猜。

而猜测会不断叠加，最终形成彻底混乱。

### 修复方法不是更花哨的提示词

真正的修复方法是：

- 更好的理解
- 更好的结构
- 更明确的上下文

当你真的知道自己在构建什么之后，提示词反而不再是重点。

因为那时候，你终于知道该问什么、该约束什么、该让 AI 做什么。

---

# 二、文档优先系统（Document First）

这是大多数人最容易做错的地方。

很多人的典型做法：

1. 打开 Cursor / Claude / Copilot
2. 打开聊天框
3. 直接开始描述 App
4. 让 AI 立刻写代码

没有计划，没有文档，没有统一真相来源。

这就是为什么项目通常在最开始几个文件之后就开始崩。

## 正确顺序：文档第一，代码第二

在写任何一行代码之前，先写清楚你的项目规范文档（Markdown 文件）。

这些文档要做到：

- 清晰
- 具体
- 明确
- 可引用
- 可被 AI 反复读取

### 为什么必须这样做？

因为 AI 编程工具：

- **能力高**，但**确定性低**
- 没有约束时，会自己脑补需求
- 会擅自做架构决策
- 会解决你从没明确提出的问题

所以真正的失败模式不是“不会写代码”，而是：

> **缺乏纪律 + 缺乏上下文保存。**

---

## 你应该先写好的 6 份规范文档

这些文档共同组成你的 **项目知识库**。

### 1. `PRD.md`
**产品需求文档（Product Requirements Document）**

定义：

- 你在构建什么
- 为谁构建
- 具体功能有哪些
- 哪些内容在范围内
- 哪些内容明确不做
- 用户故事
- 成功标准
- 非目标（non-goals）

这是整个项目的“合同”。

如果没有 PRD，你不是在构建产品，而是在祈祷产品凭空出现。

---

### 2. `APP_FLOW.md`
**用户流程文档**

记录：

- 每个页面
- 每条用户导航路径
- 每个流程的触发条件
- 成功时发生什么
- 错误时发生什么
- 屏幕清单和路由关系

它的作用是：

> 防止 AI 自己猜用户该怎么在产品里移动。

---

### 3. `TECH_STACK.md`
**技术栈文档**

把所有技术选择锁死到明确版本：

- 框架
- 依赖
- 包管理工具
- API
- 第三方服务

例如：

- Next.js 14.1.0
- React 18.2.0
- TypeScript 5.3.3

这份文档的意义在于：

- 消除歧义
- 避免 AI 自选版本
- 防止出现幻觉依赖

---

### 4. `FRONTEND_GUIDELINES.md`
**前端设计系统文档**

要定义：

- 字体
- 调色板（含精确十六进制颜色）
- 间距尺度
- 布局规则
- 组件样式
- 响应式断点
- UI 库偏好

这份文档就是你的视觉规范。

没有它，AI 每做一个组件都可能像不同的人写出来的。

---

### 5. `BACKEND_STRUCTURE.md`
**后端结构文档**

定义：

- 数据库 schema
- 每张表、每列、类型和关系
- 认证逻辑
- API 端点合约
- 存储规则
- 边缘情况处理

如果你用 Supabase，这里甚至应该写清楚 SQL 结构。

作用是：

> 让 AI 根据你的蓝图建后端，而不是根据它自己的脑补来建。

---

### 6. `IMPLEMENTATION_PLAN.md`
**实施计划文档**

按步骤拆解你的项目：

- Step 1.1 初始化项目
- Step 1.2 安装依赖
- Step 1.3 建立目录结构
- Step 2.1 构建导航栏
- Step 2.2 构建首页 hero 区
- ……

原则很简单：

> **步骤越细，AI 猜得越少；AI 猜得越少，幻觉越少。**

---

## 文档之间如何配合

这 6 份文档不是孤立的，它们互相引用：

- `PRD.md` 定义功能是什么
- `APP_FLOW.md` 定义用户如何体验这些功能
- `TECH_STACK.md` 定义用什么构建
- `FRONTEND_GUIDELINES.md` 定义它长什么样
- `BACKEND_STRUCTURE.md` 定义数据如何运作
- `IMPLEMENTATION_PLAN.md` 定义按什么顺序做

这就是你的项目知识库。

---

## 两份会话持久化文件：你的“外部记忆”

### `CLAUDE.md`
这是 AI 每次会话都会优先读取的规则文件。

里面可以写：

- 技术栈摘要
- 文件命名约定
- 组件模式
- 设计系统 token
- 什么允许做
- 什么禁止做

它就像 AI 在你这个项目里的“操作手册”。

---

### `progress.txt`
这是最容易被忽略，但又极其重要的文件。

它记录：

- 已完成什么
- 正在做什么
- 接下来做什么
- 已知 bug 有哪些

因为 AI **跨会话没有记忆**。

你关闭终端、开新会话、切换分支之后，之前的上下文就没了。

所以：

> `progress.txt` 就是你在不同会话之间的桥梁。

每次做完一个功能，都要更新它。

# 三、Interrogation 系统：先把想法审问干净

在你开始写文档之前，应该先让 AI 把你的想法“拷问”一遍。

推荐提示词：

```text
在写任何代码之前，在 Planning 模式下无尽地审问我的想法。不要假设任何问题。问问题直到没有假设剩下。
```

## 为什么要这样做？

AI 会在你的“清晰度结束”的地方开始产生幻觉。

所以，**你必须先把自己的模糊想法问透。**

### AI 应该问你的问题包括：

- 这是给谁用的？
- 用户的核心动作是什么？
- 他们完成动作后会发生什么？
- 需要保存什么数据？
- 需要展示什么数据？
- 出错时发生什么？
- 成功时发生什么？
- 是否需要登录？
- 是否需要数据库？
- 是否必须在手机上可用？

如果这些问题你答不上来，说明你还没准备好开工。

---

## 示例：食谱分享 App

当你把“食谱分享 App”这个点子审问透之后，可能会得到这样一套答案：

- 用户：想保存并分享食谱的家庭厨师
- 核心行为：用户创建食谱，包含标题、食材、步骤和照片
- 结果：食谱保存到个人资料，并出现在公共 feed
- 数据结构：标题、食材数组、步骤数组、照片 URL、作者 ID、时间戳
- 展示内容：最近食谱 feed、单个食谱详情页、个人食谱集合
- 错误状态：上传失败、缺字段、无权编辑
- 登录：创建需要登录，浏览公开
- 数据库：需要 users 表和 recipes 表
- 设备：手机优先

这时候，这些答案不只是“回答问题”，而是会变成：

- `PRD.md` 的原料
- `APP_FLOW.md` 的原料
- `BACKEND_STRUCTURE.md` 的原料
- `FRONTEND_GUIDELINES.md` 的原料

接下来再让 AI 帮你产出规范文档：

```text
基于我们的审问，生成我的规范文档文件：PRD.md、APP_FLOW.md、TECH_STACK.md、FRONTEND_GUIDELINES.md、BACKEND_STRUCTURE.md、IMPLEMENTATION_PLAN.md。用我们对话中的答案作为素材。要具体且详尽。没有歧义。
```

正确顺序只有一个：

> **审问 → 文档 → 代码**

永远不要跳过。

# 四、UI vs UX

很多人嘴上经常说 UI / UX，但其实没分清。

## UI 是什么？
UI（User Interface）是：

- 颜色
- 字体
- 按钮外观
- 间距
- 视觉呈现

也就是：**它看起来怎么样。**

## UX 是什么？
UX（User Experience）是：

- 用户是否能理解怎么用
- 流程是否顺滑
- 是否会卡住
- 用起来是愉悦还是挫败

也就是：**它用起来感觉怎么样。**

## 两者的区别
你可能有：

- 很漂亮的 UI，但糟糕的 UX
- 很普通的 UI，但很清晰好用的 UX

### 和 AI 说话时要具体
- “让它看起来更好” → UI
- “让它更容易用” → UX
- “让按钮更明显” → UI + UX

---

## 一个非常实用的技巧：给 AI 看截图

不要只用文字描述设计。

你可以：

- 找一个喜欢的网站/App
- 截图
- 直接发给 AI
- 让它“匹配这个布局”或“用这个 UI 做灵感”

AI 可以从图中提取：

- 布局模式
- 间距
- 调色板
- 排版
- 组件结构

这通常比纯文字描述强太多。

你也可以把自己当前页面截图给 AI，然后说：

- 这是现在的样子
- 问题是什么
- 帮我修掉

视觉参考能极大减少歧义。

---

## 常见设计风格速查

### 1. Glassmorphism（玻璃拟态）
特点：

- 半透明
- 背景模糊
- 微妙边框
- 柔和阴影

适合：

- 卡片
- 模态框
- 导航栏
- 仪表盘

风险：可读性可能下降。

---

### 2. Neobrutalism（新粗野主义）
特点：

- 高对比色
- 粗黑边框
- 扁平阴影
- 古怪但有个性

适合：

- 创意品牌
- 独立产品
- 作品集

---

### 3. Neumorphism（软 UI）
特点：

- 看起来像从背景里“鼓起来”或“压下去”
- 柔和阴影
- 很有触感

缺点：低对比度可能影响可访问性。

---

### 4. Bento Grid（便当网格）
特点：

- 不同大小的模块化卡片布局
- 节奏感强
- 响应式友好

适合：

- 仪表盘
- 产品页
- 功能展示页
- 作品集

---

### 5. Dark Mode（暗黑模式）
不是一个单独功能，而是完整设计系统的一部分。

要提前规划：

- 暗色背景
- 浅色文字
- 对比度
- 强调色
- 明暗两套调色板

不要最后临时加。

---

### 6. Kinetic Typography（动态排版）
特点：

- 会动的标题
- 滚动时变化的文字
- 更强的视觉戏剧性

适合 hero 区域和重点时刻。

---

### 7. Micro-interactions（微交互）
特点：

- 按钮 hover 微缩放
- 点击有反馈
- 动画细节让界面显得更“活”

这类小细节非常区分专业产品和业余产品。

---

## 给 AI 的设计描述要具体
不要说：

- “做得现代一点”

要说：

- “玻璃拟态卡片 + Bento Grid 布局 + 暗黑模式 + hover 微交互”

这是一个可执行的设计方向。

---

# 五、组件（Components）

组件就是可复用的界面片段。

你可以把它想成乐高积木。

常见组件：

- 按钮
- 导航栏
- 卡片
- 表单
- 输入框
- 标签
- 模态框

## 为什么组件思维很重要？

因为你如果只是跟 AI 说：

- “给我做个落地页”

它就得自己猜：

- 需要哪些模块？
- 怎么拆？
- 是不是全部写成一坨？

更好的说法是：

> “用这些组件构建落地页：导航栏、hero 区、功能卡片网格、推荐轮播、CTA 区域、页脚。”

这样 AI 知道要拆出哪些积木，而不是糊成一坨。

---

# 六、布局（Layout）

布局就是页面元素如何摆放。

网页本质上就是：

> **盒子套盒子。**

常见布局盒子：

- 顶部导航（Header / Navbar）
- 主内容区（Main Content）
- 侧边栏（Sidebar）
- 页脚（Footer）

主内容区里面还会继续拆：

- Section
- Container
- Grid
- Card

## 给 AI 描述布局时要明确
例如：

> “两列布局，左边固定侧边栏 250px，右边内容区占满剩余空间，侧边栏不滚动。”

这就是一条高质量布局指令。

---

# 七、State（状态）

State 就是会变化的数据。

一切“会变化”的 UI，都离不开 state。

例如：

- 菜单是开还是关
- 用户是否登录
- 购物车里有哪些商品
- 输入框里写了什么
- 正在加载还是已完成
- 提交成功还是失败

## 为什么它重要？

如果按钮点了没反应，很多时候不是按钮坏了，而是：

> **点击触发了，但没有 state 更新。**

## 和 AI 描述交互时要说清状态变化
例如：

> “用户点击按钮时，把 modal 的 state 设为 open；点击遮罩层时设回 closed。”

这样 AI 才知道它需要跟踪什么、何时更新。

---

# 八、Styling（样式）

样式决定页面“看起来怎么样”。

CSS 控制：

- 颜色
- 间距
- 字号
- 位置
- 字体
- 阴影
- 圆角

## Tailwind 是什么？
Tailwind 是一种更快捷的样式写法。

例如：

- `bg-blue-500`
- `text-xl`
- `p-4`

不用写一堆单独 CSS 文件，直接在类名里表达样式。

## Design Tokens（设计令牌）
设计令牌的核心思想是：

- 品牌蓝就固定成同一个 hex
- 圆角有统一值
- 阴影有统一值
- 间距按固定尺度走

这样整个产品才会有一致性。

如果每个组件都临时发明自己的颜色和间距，最终看起来就像 5 个人拼出来的。

---

# 九、Responsive Design（响应式设计）

响应式的意思是：

> **同一个产品，在不同屏幕尺寸上都能正常工作。**

## 典型场景
- 大屏显示多列
- 小屏改成单列堆叠
- 桌面显示完整导航
- 手机改成汉堡菜单

## 常见断点
- 手机：0–640px
- 平板：640–1024px
- 桌面：1024px+

## Mobile-first（移动优先）
这是非常重要的策略。

先为手机设计，再为大屏加复杂度。

原因：

- 手机用户往往更多
- 小屏会逼你保留真正重要的内容
- 有助于形成更克制、更清晰的 UX

在前端规范里应该把这些写死，例如：

- 导航：768px 以下汉堡菜单，以上横向导航
- 网格：手机 1 列，平板 2 列，桌面 3 列
- 字体：随断点逐步放大

# 十、Pages vs Routes

- **Page**：用户看到的页面
- **Route**：显示这个页面的 URL

例如：

- `/` → 首页
- `/about` → 关于页
- `/products/123` → 商品 123 页面

路由可能是：

- 静态路由：`/about`
- 动态路由：`/products/[id]`

和 AI 说时要明确页面结构，比如：

> “我需要四个页面：首页、关于页、商品列表页、商品详情页。”

---

# 十一、Frontend vs Backend

## Frontend
用户看得见、点得到的部分。

运行在浏览器里。

## Backend
幕后系统：

- 数据库
- 用户认证
- 业务逻辑
- 服务器处理

### 表单提交流程举例
1. 前端收集输入
2. 发给后端
3. 后端验证并存储
4. 后端返回结果
5. 前端显示成功/失败提示

如果只是静态展示页，可能不需要后端。

如果有：

- 登录
- 数据保存
- 用户内容
- 支付

那就需要后端。

---

# 十二、API

API 就是系统之间的沟通接口。

你可以把它想成“服务员”：

- 前端点单
- 后端/外部服务做饭
- 再把结果端回来

常见请求类型：

- `GET`：读取数据
- `POST`：提交数据
- `PUT`：更新数据
- `DELETE`：删除数据

示例提示：

> “页面加载时，请向 `/api/products` 发起 GET 请求，并把结果展示到网格里。”

---

# 十三、数据库（Databases）

数据库是让数据在刷新后依然存在的地方。

没有数据库，你刷新一次：

- 用户没了
- 购物车没了
- 发帖没了

## 初学者推荐：Supabase
原因：

- 上手简单
- 文档友好
- 免费层可用
- 还自带认证能力

## 什么时候需要数据库？
- 用户可注册
- 用户可保存内容
- 有增长中的数据
- 需要持久化记录

## 什么时候不需要？
- 静态作品集
- 单纯展示页
- 没有用户内容的 landing page

示例：

> “用 Supabase。我需要一个 users 表和一个 posts 表，posts 包含 title、content、user_id。”

---

# 十四、认证（Authentication）

认证就是：

- 注册
- 登录
- 登出
- 证明某个用户是谁

这是很容易出问题的部分，因此：

> **不要自己从零做 auth。用成熟服务。**

常见方案：

- Clerk
- Supabase Auth

示例：

> “使用 Clerk 做认证，支持 email 注册和 Google 登录，登录后跳转到仪表盘。”

---

# 十五、常见文件类型速查

- `.html`：网页结构
- `.css`：样式规则
- `.js`：JavaScript 逻辑
- `.jsx`：带 JSX 的 React 代码
- `.ts`：TypeScript
- `.tsx`：React + TypeScript
- `.json`：结构化数据
- `.md`：Markdown 文档
- `.env`：环境变量和密钥
- `.gitignore`：Git 忽略规则

## 特别提醒：`.env`
`.env` 里常放：

- API Key
- 数据库密码
- Token
- Secret

规则非常简单：

- 永远不要分享
- 永远不要提交到 git
- 永远不要截图给别人

一旦泄露，默认它已经失控。

# 十六、目录结构（Folder Structure）

好的结构会让 AI 更稳定，坏结构会让 AI 更混乱。

典型目录：

```text
my-app/
├── src/
│   ├── app/           → 页面和路由
│   ├── components/    → 可复用 UI 片段
│   ├── lib/           → 工具函数
│   └── styles/        → 样式文件
├── public/            → 静态资源
├── .env               → 密钥（不外传）
├── CLAUDE.md          → AI 规则文件
├── progress.txt       → 会话跟踪
├── PRD.md
├── APP_FLOW.md
├── TECH_STACK.md
├── FRONTEND_GUIDELINES.md
├── BACKEND_STRUCTURE.md
├── IMPLEMENTATION_PLAN.md
├── package.json
└── README.md
```

和 AI 沟通时，最好明确说：

> “把按钮组件写到 `src/components/Button.tsx`。”

否则它可能随手乱放文件。

# 十七、文档系统在实践中怎么用

Markdown 不是可选项，而是 AI 理解项目的语言。

这些文件在实际开发中的作用：

- `PRD.md`：定义范围，阻止 scope creep
- `APP_FLOW.md`：定义用户流程和跳转
- `TECH_STACK.md`：阻止 AI 引入未授权依赖
- `FRONTEND_GUIDELINES.md`：保证视觉一致性
- `BACKEND_STRUCTURE.md`：保证数据层符合预期
- `IMPLEMENTATION_PLAN.md`：保证执行顺序正确
- `CLAUDE.md`：统一规则总入口
- `progress.txt`：跨会话持续上下文

---

## `CLAUDE.md` 其实是活文档

每次 AI 犯错并被你纠正后，都应该补一句：

> “编辑 `CLAUDE.md`，避免你以后再犯同样的错。”

因为 Claude 很擅长给自己写规则。

随着纠错积累，`CLAUDE.md` 会越来越像一个项目级“自我修正手册”。

你还可以额外增加：

### `lessons.md`
记录：

- 出过什么问题
- 是什么模式导致的
- 后续应该怎么避免

然后在 `CLAUDE.md` 里写明：

- 每次会话开始先读 `lessons.md`
- 每次纠错后更新 `lessons.md`

这就形成了一个真正的自我提升闭环。

# 十八、工具以及如何使用它们

你不需要 50 个工具，但你确实需要几个核心工具，并且要知道不同阶段用哪个。

## 1. Cursor
主要开发编辑器。

它常见的 4 种模式：

### Ask Mode
只读，不改代码。

适合：

- 理解项目
- 读陌生代码
- 提前问清逻辑

### Plan Mode
先做实施计划。

适合：

- 新功能开始前
- 大改之前
- 架构设计

### Agent Mode
真正自动改代码、跑命令、修问题的主力模式。

### Debug Mode
结构化调试模式。

适合：

- 顽固 bug
- 需要日志、假设、验证链路的问题

### 内部工作流
> Ask 理解 → Plan 规划 → Agent 构建 → Debug 修错

---

## 2. Claude / Claude Code
适合：

- 审问需求
- 写规范文档
- 规划架构
- 处理复杂产品决策
- 做大规模重构和多文件变更

Claude Code 会自动读取项目根目录下的 `CLAUDE.md`。

---

## 3. Kimi K2.5
Moonshot 的视觉编码模型。

特别适合：

- 前端视觉还原
- 根据截图生成 UI
- 布局、动画、交互实现

如果你的任务是视觉型 frontend，实现精度它很强。

---

## 4. Codex
更适合：

- 调试
- 代码审查
- 重构
- 跑测试直到通过
- 发现你没注意到的问题

你可以把它当成项目后期的“终结者”。

---

## 推荐多工具工作流
- Claude：负责思考和文档
- Cursor Agent / Claude Code / Kimi：负责构建
- Codex：负责调试、审查、收尾

---

## 5. GitHub
代码云端仓库 + 版本控制。

作用：

- 每次变更有记录
- 可以回滚
- 可以协作
- 可以部署对接

---

## 6. Vercel
部署平台。

常见流程：

1. 推送代码到 GitHub
2. Vercel 连接仓库
3. 自动构建和部署
4. 生成可访问网址

---

## 7. Supabase
适合做：

- 数据库
- 认证
- 存储

对独立开发非常友好。

# 十八点五、高级工作流

当你已经掌握基础之后，可以继续进阶。

## 1. Git worktrees 并行开发
一次开多个 worktree：

- 一个做认证
- 一个做仪表盘
- 一个做 API

并行推进，再合并。

适合大型项目。

---

## 2. 用 Plan Mode 做更强的验证
不仅用来“怎么做”，还可以用来“怎么验证”。

例如：

- 如何验证登录流程覆盖所有边缘情况？
- 如何验证结账流程不会重复扣费？

---

## 3. 子代理（Sub-agents）
适合复杂问题：

- 一个子代理查资料
- 一个子代理看代码
- 一个子代理找 bug

主会话保持简洁，子会话分头做事。

---

## 4. 自定义技能与斜杠命令
如果某件事你每天都做，不要重复手工操作。

把它固化成：

- 可复用命令
- skill
- 脚本

例如：

- `/techdebt`
- `context-sync`
- `release-check`

---

## 5. 自主 bug 修复
不要每次都手把手带 AI 调 bug。

可以直接给：

- bug 描述
- 日志
- CI 报错
- 失败测试

让 AI 自己去追踪根因并修复。

---

## 6. 语音听写
说话通常比打字快 3 倍。

更长、更细节、更自然的提示词，经常会带来更好的结果。

---

## 7. 学习模式
不要只让 AI 帮你做，也要让它解释它为什么这么做。

长期看，这会让你真正从“只会 vibe”升级到“懂得构建”。

# 十九、Git 和版本控制

Git 的作用：

- 跟踪每次更改
- 允许回滚
- 避免一处失误毁掉所有成果

## 基础命令
- `git add .`
- `git commit -m "message"`
- `git push`
- `git pull`

## 开发建议
每做完一个稳定功能就提交一次。

例如：

- `add user login`
- `build product grid`
- `fix checkout bug`

这跟 `progress.txt` 是互补的：

- Git 保存代码状态
- `progress.txt` 保存项目语义状态

---

# 二十、环境变量和密钥

核心原则：

- API key 不应写进前端代码
- 密钥应存在 `.env`
- 不要把 `.env` 提交到 git
- 生产环境和开发环境要分开

部署到 Vercel 时，别忘了在平台设置里单独配置环境变量。

如果泄露了 key：

> 立即撤销并重建。

---

# 二十一、部署

本地能跑，不等于线上能跑。

Vercel 的基本流程：

1. 推代码到 GitHub
2. 仓库接入 Vercel
3. 自动构建和部署
4. 配置环境变量
5. 获取线上 URL

如果出现“本地正常，线上崩了”，高概率是：

- 环境变量没配
- 构建设置不对
- 线上环境差异导致

---

# 二十二、阅读错误信息

错误不是对你的羞辱，而是指令。

例如：

```text
TypeError: Cannot read property 'map' of undefined
at ProductList (src/components/ProductList.tsx:15:23)
```

这实际上已经告诉你：

- 错误类型：TypeError
- 原因：对 `undefined` 调用了 `map`
- 文件：`ProductList.tsx`
- 位置：第 15 行第 23 列

## 正确做法
把完整错误 + 相关代码 + 你的预期行为一起给 AI。

上下文越完整，修得越快。

---

# 二十三、调试循环

当东西坏了，调试流程应该是：

1. 认真读错误
2. 找到对应文件和行号
3. 理解这条错误在说什么
4. 先检查最明显的问题
5. 把完整上下文交给 AI

调试本来就是迭代过程：

- AI 给修复
- 你尝试
- 又报错
- 把新错误贴回去
- 再修
- 直到通过

这不是失败，这是正常开发节奏。

---

# 二十四、发布前验证清单

上线前至少检查：

- [ ] 手机上真的能用吗？
- [ ] 不同浏览器能跑吗？
- [ ] 空状态处理了吗？
- [ ] 错误状态处理了吗？
- [ ] 慢网速时有 loading 吗？
- [ ] 快速乱点会不会崩？
- [ ] 密钥不会暴露在浏览器开发者工具里吗？

如果这些没测，用户会替你测，而且通常是以一种很难看的方式。

---

# 二十五、如何跟 AI 说话

## 糟糕的提示
> 给我构建一个用户可以发帖的 App

## 高质量提示
> 首先读取 `CLAUDE.md` 和 `progress.txt`。然后构建 `IMPLEMENTATION_PLAN.md` 的步骤 4.2。登录流程在 `APP_FLOW.md` 第 3 节定义。认证配置使用 `BACKEND_STRUCTURE.md` 第 5 节。界面风格遵循 `FRONTEND_GUIDELINES.md`。UI 参考附加截图。

同样的任务，输出质量会完全不同。

### 结论
> **具体性不是附加项，具体性本身就是工作。**

---

# 二十六、如何阅读 AI 输出

你不一定要懂每一行语法，但你至少要搞清这些：

- 它创建了哪些文件
- 每个文件负责什么
- 文件之间怎么连接

一个很好用的问题是：

> “请用 plain English 解释你刚刚构建了什么。每个文件做什么？它们如何连接？”

慢慢地，你会开始识别模式，逐步真正理解项目结构。

---

# 二十七、如何迭代

第一次输出几乎从来都不是最终正确版本。

正常迭代流程：

1. AI 构建 v1
2. 你测试
3. 明确指出哪里不对
4. AI 修复
5. 继续测试
6. 重复

## 好的反馈示例
- 桌面端产品网格现在是 4 列，我需要 3 列
- 图片被拉伸了，应该改成 `object-cover`
- 数据加载时没有 loading state

## 坏的反馈示例
- 看起来不对，修一下

一定要具体。

---

# 二十八、把大想法拆成小块

AI 很容易被“大而空”的需求噎住。

糟糕需求：

> “给我做一个完整电商网站。”

更好的拆法：

1. 初始化项目脚手架
2. 安装依赖
3. 构建导航栏
4. 构建产品卡片
5. 构建产品列表页
6. 连接数据库
7. 构建商品详情页
8. 加入购物车
9. 构建购物车页面
10. Stripe 结账

这其实就是 `IMPLEMENTATION_PLAN.md` 应该长成的样子。

你不是在写代码时临时想步骤，而是在开始前就拆好。

---

# 二十九、什么时候 AI 不是对的工具

有些东西你最终还是得自己学。

AI 很适合：

- 样板代码
- 重复逻辑
- 快速探索方案
- 带上下文的调试
- 把意图翻译成代码

你自己必须掌握：

- 基础概念
- 如何读 AI 生成代码
- 如何识别 AI 出错了
- AI 帮不上忙时怎么 debug
- 你所用技术栈的基本工作原理

另外，官方文档和 Stack Overflow 往往比 AI 更权威。

AI 很强，但不是神。

---

# 三十、范围控制与知道何时停止

很多项目不是死在 bug 上，而是死在：

> **功能永远加不完。**

## 什么时候算完成？
- 核心功能能跑
- 用户能完成主要行为
- 常见路径不崩
- 已部署并可访问

## 什么时候不算完成？
- 你还想再加 10 个功能
- 你还在追求“看起来像 Dribbble 那样完美”
- 你还没发布第一版

先发布一个简单但可用的版本，再基于真实反馈改进。

---

# 三十一、维护你构建的东西

你构建完后，未来还要改。

未来的你也需要读得懂。

所以：

- README 解释项目是什么
- `CLAUDE.md` 约束 AI 行为
- `progress.txt` 记录状态
- 各种规范文档保证项目可追踪

当你三个月后回到项目时，应该能这样启动：

> “读取 `CLAUDE.md`、`progress.txt` 和 `PRD.md`。我刚回到这个项目。根据 `IMPLEMENTATION_PLAN.md` 总结当前进度和需要注意的地方。”

好的文档能让你重新上手速度提升很多。

---

# 三十二、成本意识

AI、托管、数据库、API 都可能花钱。

幸运的是，免费层通常已经足够你起步。

原则：

- 从免费层开始
- 有明确需求再升级
- 不要第一天就按百万用户规格设计系统

现实是：

> 你大概率还没到那个量级。

先把东西做出来。

---

# 三十三、安全基础

最低要求：

- 不要把 API key 放前端
- 后端必须验证输入
- 使用 HTTPS
- 依赖保持更新
- 认证优先用成熟服务

你不需要把产品做成银行级安全系统，
但也别把它做成一个公开邀请别人攻击的玩具。

---

# 三十四、第三方服务

不是所有东西都要自己造。

适合直接接第三方服务的部分：

- 认证：Clerk / Supabase Auth
- 支付：Stripe
- 数据库：Supabase / Firebase
- 邮件：Resend / SendGrid
- 文件上传：Uploadthing / Supabase Storage

原则：

> **凡是不属于你产品核心竞争力的基础设施，尽量交给成熟服务。**

---

# 三十五、素材、媒体与组件库

## 素材来源
- 图标：Lucide React
- 字体：Google Fonts
- 图片：Unsplash / 自己素材 / AI 生成

记住：图片体积很重要，太大会拖慢页面。

## 组件库：新手大加速器

不要从零开始重复造按钮、对话框、下拉框、表单。

推荐：**shadcn/ui**

优点：

- 组件质量高
- 可访问性不错
- 可定制
- 直接把源代码复制进项目，你自己拥有它

在 `TECH_STACK.md` 和 `FRONTEND_GUIDELINES.md` 中明确写清你选用的组件库，AI 才会全项目一致地使用它。

# 完整系统总结

## 开始之前
- 审问想法
- 回答所有关键问题
- 生成六份规范文档
- 写好 `CLAUDE.md`
- 建立 `progress.txt`
- 准备 UI 参考截图
- 初始化 git 并推送 GitHub

## 构建中
- 每次会话优先读取 `CLAUDE.md`、`progress.txt`、`lessons.md`
- 用 Ask / Plan 做规划
- 用 Agent / Claude Code / Kimi 做实现
- 一次只做一个小块
- 用具体词汇和规范文档约束 AI
- 每完成一个功能就 git commit
- 更新 `progress.txt`
- 每次纠错后更新 `CLAUDE.md` 与 `lessons.md`
- 适时用 Codex 调试和审查
- 定期手机实测

## 发布前
- 检查手机适配
- 检查空状态、错误状态
- 检查密钥安全
- 跑通主要流程
- 看性能问题

## 发布后
- 文档同步更新
- 依赖定期更新
- 按真实用户反馈迭代
- 继续维护 `progress.txt` 和 `lessons.md`
- 把重复流程工具化
