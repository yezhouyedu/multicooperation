\# 实验随机化设计说明



\## 一、随机化设计的总体目标



本实验采用实验室现场实验形式，参与者以两人团队完成投资尽调任务。实验的核心目标是识别人工智能大模型能力、注意力环境和合作叙事如何影响上下游团队生产链中的信息提取、信息承接、判断整合和最终团队绩效。



随机化设计需要同时满足四个要求。



第一，处理赋予必须与参与者到场时间、登录顺序、座位顺序和实验员操作相隔离。实验室现场必然存在参与者先后到达、先后登录、预报名但未到、迟到和替补等情况，因此预报名名单不能直接参与处理随机化。



第二，A/B 角色必须随机分配。A 与 B 是团队生产链中的两个固定位置。A 是上游信息供给者，B 是下游判断整合者。角色不能由到场顺序、座位位置或登录先后决定。



第三，三个实验局的处理均随机到团队层面。A 和 B 共享同一条生产链，A 的输出会影响 B 的判断，B 的反馈也可能影响 A 后续行为。若在个人层面随机，容易造成团队内污染。



第四，材料分配、新闻顺序和公司推送机制需要和实验处理随机化区分开。宏观/微观材料分配属于公司材料层面的预设随机化；新闻 ID 顺序属于呈现顺序随机化；公司推送机制属于统一流程规则。它们都需要记录，但不等同于三个实验局的主处理变量。

\## 二、随机化层级总览



本实验包含五个层级的随机化或准随机安排。



| 层级      | 随机对象                              | 是否主处理变量   | 实现方式                           |

| ------- | --------------------------------- | --------- | ------------------------------ |

| 参与者层    | 实际到场并登录的参与者随机匹配为团队                | 否         | 同一启动批次内服务器随机匹配                 |

| 团队内部    | A/B 角色随机分配                        | 否         | team\_id 形成后服务器随机分配             |

| 团队层     | 三个实验局的处理状态                        | 是         | team\_id 形成后领取隐藏 treatment slot |

| 合作叙事组内部 | 三类合作叙事主题在三个工作段中的顺序                | 嵌套随机化     | 合作叙事组内随机分配六种排列之一               |

| 公司层     | A\_macro\_B\_micro / A\_micro\_B\_macro | 否，材料预设随机化 | 材料制作阶段固定到 company\_id           |

| 新闻流层    | news\_id 在工作段内的展示顺序                | 否，呈现顺序随机化 | 团队层面随机，团队内 A/B 相同              |

| 公司推送    | B 接收下一家公司的流程随机                    | 否，统一流程规则  | 锁定池优先，池内随机；必要时 B\_PreA 预启动      |

\## 三、实验室现场进入与匹配规则



\### 1. 预报名名单不参与处理随机化



预报名名单只用于排期和现场管理。预报名编号不直接绑定 A/B 角色，也不直接绑定实验处理。



预报名阶段可记录：



| 字段                   | 含义       |

| -------------------- | -------- |

| registered\_id        | 预报名编号    |

| scheduled\_session\_id | 预约场次     |

| scheduled\_time       | 预约时间     |

| contact\_status       | 联系状态     |

| no\_show\_flag         | 是否预报名但未到 |

| late\_arrival\_flag    | 是否迟到     |

| standby\_flag         | 是否替补     |



预报名但实际未到的参与者不生成正式 participant\_id，不进入匹配池，不占用 treatment slot。



\### 2. 实际到场并登录后进入待匹配池



参与者到实验室后完成签到和平台登录。只有实际到场并完成登录的参与者进入本场次待匹配池。



系统生成：



| 字段               | 含义        |

| ---------------- | --------- |

| participant\_id   | 正式实验参与者编号 |

| registered\_id    | 预报名编号，可空  |

| session\_id       | 实验场次      |

| check\_in\_time    | 现场签到时间    |

| login\_time       | 平台登录时间    |

| terminal\_id      | 终端或座位编号   |

| arrival\_batch\_id | 启动批次编号    |

| standby\_flag     | 是否替补      |



\### 3. 同一启动批次内随机匹配团队



由于参与者不会完全同时登录，系统不应简单按照登录顺序两两配对。推荐做法是设置启动批次。



例如：



第一批次：正式开始前 0—10 分钟内完成登录的参与者。

第二批次：迟到但仍可进入实验的参与者，或下一轮启动参与者。



在同一启动批次内，服务器将待匹配参与者随机打乱后两两配对，生成 team\_id。



若某一批次人数为奇数，则采用以下规则：



1\. 系统随机保留 1 人进入等待池；

2\. 若有替补参与者，则由替补补齐；

3\. 若无法补齐，该参与者不进入正式实验，可安排下一场或支付到场补偿；

4\. 不允许单人进入正式主实验。



记录字段：



| 字段               | 含义          |

| ---------------- | ----------- |

| matching\_pool\_id | 匹配池编号       |

| arrival\_batch\_id | 启动批次        |

| team\_id          | 团队编号        |

| participant\_1\_id | 团队成员 1      |

| participant\_2\_id | 团队成员 2      |

| pairing\_seed     | 匹配随机种子或算法版本 |

| pairing\_time     | 匹配完成时间      |

| unmatched\_flag   | 是否未能匹配      |

| unmatched\_reason | 未匹配原因       |



\### 4. 团队形成后随机分配 A/B 角色



team\_id 形成后，服务器在团队内部随机分配 A/B 角色。A/B 角色一经分配，在整个实验中保持不变。



记录字段：



| 字段                     | 含义                |

| ---------------------- | ----------------- |

| team\_id                | 团队编号              |

| participant\_A\_id       | 被分配为 A 的参与者       |

| participant\_B\_id       | 被分配为 B 的参与者       |

| role\_assignment\_seed   | 角色随机种子或算法版本       |

| role\_assignment\_time   | 角色分配时间            |

| manual\_override\_flag   | 是否人工覆盖，原则上为 false |

| manual\_override\_reason | 人工覆盖原因，可空         |



A 是上游信息供给者，B 是下游判断整合者。A/B 不是固定的“宏观材料角色”或“微观材料角色”。某家公司中，A 可能收到宏观材料，也可能收到微观材料；B 同理。宏观/微观材料由公司层面的 material\_assignment\_pattern 决定。

\## 四、团队层面的处理 slot 分配



\### 1. 处理 slot 只在 team\_id 形成后消耗



处理分配发生在 team\_id 形成并完成 A/B 角色分配之后。no-show 不占用 slot；未成功匹配成团队的参与者不占用 slot；正式实验前作废的团队原则上不进入正式样本，并记录作废原因。



基本逻辑是：



```text

实际到场并登录 → 进入匹配池

同一批次内随机匹配 → 生成 team\_id

team\_id 内随机 A/B → 领取隐藏 treatment slot

```



\### 2. 场次内隐藏随机 slot



每一场实验开始前，系统为该场次生成隐藏随机 slot 列表。每个 slot 预先绑定一个 treatment\_assignment。团队形成后领取下一个未使用 slot。



为了防止 no-show 或迟到导致处理组不平衡，推荐使用小区组随机或随机区组大小。例如，实验局 2 可在每个区组中包含相同数量的 batch 和 continuous slot，再随机排列。



记录字段：



| 字段                     | 含义                |

| ---------------------- | ----------------- |

| audit\_id               | 随机化审计记录编号         |

| session\_id             | 实验场次              |

| arrival\_batch\_id       | 启动批次              |

| team\_id                | 团队编号              |

| experiment\_chapter     | 实验局编号             |

| treatment\_assignment   | 团队处理状态            |

| slot\_id                | 隐藏随机 slot         |

| randomization\_block    | 场次内随机区组           |

| assignment\_time\_server | 处理分配时间            |

| randomizer\_version     | 随机化服务版本           |

| manual\_override\_flag   | 是否人工覆盖，原则上为 false |

| manual\_override\_reason | 人工覆盖原因，可空         |

\## 五、实验局 1：AI 能力升级随机化



\### 1. 设计类型



实验局 1 是被试内设计与团队层面随机化错位升级的结合。



被试内设计体现在：同一个团队在不同工作段中会经历基础版 AI 和升级版 AI。



团队层面随机化体现在：团队被随机分配到早升级队列或晚升级队列。



因此，实验局 1 不是永久处理组和永久控制组，而是随机化错位升级设计。



\### 2. 处理组定义



| 团队类型          | 工作段 1  | 工作段 2  | 工作段 3  |

| ------------- | ------ | ------ | ------ |

| early\_upgrade | 基础版 AI | 升级版 AI | 升级版 AI |

| late\_upgrade  | 基础版 AI | 基础版 AI | 升级版 AI |



所有团队最终都会升级。第二工作段提供最清晰的升级 vs 尚未升级的同期比较；第三工作段用于观察升级后的持续效应和适应过程。



\### 3. AI 升级内容



本版 AI 升级至少包括两项能力变化：



| 能力项     | 基础版 AI | 升级版 AI     |

| ------- | ------ | ---------- |

| AI 图片上传 | 不可上传图片 | 可向 AI 上传图片 |

| 模型规模    | 小参数模型  | 全参数模型      |



开发者需要注意，AI 升级是团队层面同步发生的。团队内 A 和 B 在任一时点使用同一 AI 能力版本。



\### 4. 必须记录的变量



| 字段                                 | 含义                           |

| ---------------------------------- | ---------------------------- |

| upgrade\_cohort                     | early\_upgrade / late\_upgrade |

| segment\_1\_ai\_state                 | 工作段 1 的 AI 状态                |

| segment\_2\_ai\_state                 | 工作段 2 的 AI 状态                |

| segment\_3\_ai\_state                 | 工作段 3 的 AI 状态                |

| ai\_state\_at\_request                | 每次 AI 请求发起时的 AI 状态           |

| ai\_model\_version\_at\_request        | 请求发起时模型版本                    |

| ai\_image\_upload\_enabled\_at\_request | 请求发起时是否可上传图片                 |

| A\_ai\_state\_at\_window               | A 处理该公司时的 AI 状态              |

| B\_preA\_ai\_state                    | B\_PreA 阶段 AI 状态              |

| B\_postA\_ai\_state                   | B\_PostA 阶段 AI 状态             |

| cross\_upgrade\_boundary\_flag        | 公司流程是否跨升级边界                  |



同一家公司可能经历多个阶段：A\_window、B\_PreA 和 B\_PostA。若这些阶段跨越 AI 升级节点，则必须分别记录阶段级 AI 状态，不能只记录团队在实验结束时是否已升级。

\## 六、实验局 2：副线到达节奏随机化



\### 1. 设计类型



实验局 2 是团队层面的组间随机实验。团队一旦被分配到某种副线任务到达方式，整个实验局内都保持不变。



\### 2. 处理组定义



| 团队类型       | 工作段 1      | 工作段 2      | 工作段 3      |

| ---------- | ---------- | ---------- | ---------- |

| continuous | continuous | continuous | continuous |

| batch      | batch      | batch      | batch      |



continuous 表示副线任务高频逐条到达。

batch 表示副线任务低频分批到达。



\### 3. 必须保持一致的条件



两组除副线到达节奏外，其余条件应保持一致：



\* AI 能力；

\* 主线流程；

\* 叙事信息；

\* 副线任务总量；

\* 副线任务难度；

\* 副线任务奖励；

\* 副线任务段内保留、段末归档规则；

\* 副线 AI 功能；

\* 顶部提示方式。



顶部提示只显示“有新副线任务到达”等中性提示，不显示“当前未处理 X 条”。



\### 4. 批处理行为



参与者可能把多条副线新闻复制到副线 AI 中统一处理。该行为不视为违规，也不作为污染剔除，而是 batch 工作流组织方式的一部分。系统需要记录疑似批处理行为。



记录字段：



| 字段                                 | 含义                 |

| ---------------------------------- | ------------------ |

| side\_dispatch\_mode                 | continuous / batch |

| side\_item\_id                       | 副线任务 ID            |

| side\_item\_release\_time             | 副线释放时间             |

| side\_item\_open\_time                | 副线打开时间             |

| side\_item\_submit\_time              | 副线提交时间             |

| side\_ai\_thread\_id                  | 副线 AI 对话线程         |

| side\_ai\_prompt\_text                | 副线 AI 输入           |

| side\_ai\_response\_text              | 副线 AI 输出           |

| paste\_event\_in\_side\_ai             | 副线 AI 输入框是否发生粘贴    |

| side\_ai\_bulk\_candidate             | 是否疑似批处理            |

| bulk\_item\_count\_inferred           | 推测一次处理涉及多少条副线任务    |

| side\_items\_archived\_at\_segment\_end | 工作段结束时归档的副线任务      |

\## 七、实验局 3：合作叙事随机化



\### 1. 设计类型



实验局 3 是团队层面的组间随机实验。团队被随机分配到合作叙事组或中性信息组。



\### 2. 主处理组定义



| 处理状态           | 含义                            |

| -------------- | ----------------------------- |

| coop\_narrative | 整局实验接触合作叙事材料，同时在必要位置接触匹配的中性信息 |

| neutral\_info   | 整局实验只接触中性信息                   |



合作叙事组和中性信息组在信息展示位置、文本长度、新闻数量、行业相关性和情绪强度上应尽量匹配。前端不得显示“合作叙事组”“处理组”等标签。



\### 3. 合作叙事的三类主题



合作叙事包含三类主题：



| 主题编号 | 主题名称 | 内容含义                          |

| ---- | ---- | ----------------------------- |

| T1   | 互补分工 | AI 提高局部效率后，团队更需要上下游信息承接       |

| T2   | 验证留痕 | AI 并未削弱证据核验和来源透明度的重要性         |

| T3   | 共同责任 | AI 是辅助资源，不是绕过队友的捷径，团队结果需要共同负责 |



\### 4. 三主题展示顺序随机化



合作叙事组在三个工作段分别接触三类主题。每个工作段只接触一种主导主题。展示位置包括：



```text

段 1 前文本 + 段 1 新闻流

段 2 前文本 + 段 2 新闻流

段 3 前文本 + 段 3 新闻流

```



三类主题在合作叙事组内部的展示顺序随机化。服务器为每个合作叙事团队随机分配以下六种顺序之一：



| theme\_order | 段 1  | 段 2  | 段 3  |

| ----------- | ---- | ---- | ---- |

| T1\_T2\_T3    | 互补分工 | 验证留痕 | 共同责任 |

| T1\_T3\_T2    | 互补分工 | 共同责任 | 验证留痕 |

| T2\_T1\_T3    | 验证留痕 | 互补分工 | 共同责任 |

| T2\_T3\_T1    | 验证留痕 | 共同责任 | 互补分工 |

| T3\_T1\_T2    | 共同责任 | 互补分工 | 验证留痕 |

| T3\_T2\_T1    | 共同责任 | 验证留痕 | 互补分工 |



中性信息组不接触合作主题，但也应有对应的中性文本与中性新闻流版本，以保证页面位置和阅读流程一致。



\### 5. 新闻流顺序随机化



实验局 3 中，新闻流除符合干预组别和叙事主题外，新闻 ID 的展示顺序也需要随机化。



规则如下：



1\. 每条新闻具有唯一 news\_id。

2\. 合作叙事组中，新闻 ID 必须符合该工作段的主题。

3\. 中性信息组中，新闻 ID 必须来自对应的中性新闻池。

4\. 在满足组别、工作段和主题约束后，服务器为每个团队生成新闻 ID 的随机顺序。

5\. 同一团队内，A 和 B 看到相同新闻 ID 顺序。

6\. 不同团队之间，新闻 ID 顺序可以不同。

7\. 系统记录 news\_order\_seed 和 news\_sequence\_id。



\### 6. 必须记录的变量



| 字段                          | 含义                            |

| --------------------------- | ----------------------------- |

| narrative\_group             | coop\_narrative / neutral\_info |

| theme\_order                 | 合作叙事组的主题顺序                    |

| segment\_1\_theme             | 段 1 前主题                       |

| segment\_2\_theme             | 段 2 前主题                       |

| segment\_3\_theme             | 段 3 前主题                       |

| narrative\_text\_id\_segment\_1 | 段 1 前文本 ID                    |

| narrative\_text\_id\_segment\_2 | 段 2 前文本 ID                    |

| narrative\_text\_id\_segment\_3 | 段 3 前文本 ID                    |

| side\_news\_theme\_id          | 新闻所属叙事类别                      |

| news\_sequence\_id            | 新闻流随机序列编号                     |

| news\_order\_seed             | 新闻顺序随机种子                      |

| news\_id\_order\_within\_team   | 团队内新闻 ID 展示顺序                 |

| news\_id\_display\_time        | 新闻展示时间                        |

\## 八、公司材料分配的预设随机化



\### 1. 材料属性与分配端分离



材料有两个概念：材料属性和材料分配端。



| 概念    | 允许值            | 含义              |

| ----- | -------------- | --------------- |

| 材料属性  | 微观 / 宏观 / 公司背景 | 材料本身承载的信息类型     |

| 材料分配端 | A / B / 共同     | 该材料在某家公司中由谁初始查看 |

| 材料类型  | 文本 / 图片 / 表格   | 材料呈现形式          |



材料属性不自动决定分配端。微观材料可以分配给 A，也可以分配给 B；宏观材料同理。



\### 2. 两种公司材料分配 pattern



每家公司在材料制作阶段被预设为以下两种 pattern 之一：



| material\_assignment\_pattern | A 初始材料 | B 初始材料 |

| --------------------------- | ------ | ------ |

| A\_macro\_B\_micro             | 宏观材料   | 微观材料   |

| A\_micro\_B\_macro             | 微观材料   | 宏观材料   |



某家公司一旦设定为某一 pattern，在所有团队中保持一致。例如：



```text

company\_id = C001 → 永远是 A\_macro\_B\_micro

company\_id = C002 → 永远是 A\_micro\_B\_macro

```



这属于公司材料层面的预设随机化，不是团队层面的实验处理变量。

\## 九、公司推送机制与随机化的关系



公司推送机制不是三章主处理变量，而是所有实验局共用的流程规则。其作用是保证 A/B 上下游生产链持续运行，同时避免 B 因已锁定公司池暂时为空而获得额外副线时间。



\### 1. 首家公司同步启动



正式实验开始时，A 和 B 同时接收同一家公司。



A 看到该公司的 A 端材料。

B 看到该公司的 B 自有材料。

A 输出仍按统一解锁规则显示给 B。

首家公司记录 `initial\_company = 1`。



\### 2. 后续公司推送规则



B 完成当前公司报告和反馈后，系统按以下顺序处理：



| 条件                       | 操作                                       |

| ------------------------ | ---------------------------------------- |

| 已锁定公司池非空                 | 从已由 A 锁定、尚未分配给 B 的公司中随机抽取一家公司            |

| 已锁定公司池为空，但 A 正在处理下一家公司   | 将 A 当前正在处理的公司推送给 B，B 进入 B\_PreA\_pending\_A |

| 已锁定公司池为空，且 A 无 active 公司 | B 进入真实空窗状态                               |



\### 3. B\_PreA\_pending\_A 状态



若 B 接收的是 A 当前正在处理但尚未锁定的公司，B 只进入 PreA 阶段。



B 可以：



\* 查看 B 自有材料；

\* 填写信息点草稿；

\* 撰写备忘录草稿；

\* 调用仅限 B 自有材料的主线 AI；

\* 切换副线任务。



B 不能：



\* 查看 A 材料；

\* 查看 A 尽调表；

\* 查看 A 总体备注；

\* 进入 A 原始材料复核入口；

\* 最终提交该公司。



A 输出解锁仍使用统一公式：



```text

A\_output\_unlock\_time =

max(A\_window\_lock\_time, B\_receive\_time + B\_preA\_minutes)

```



\### 4. 公司推送日志



| 字段                               | 含义                   |

| -------------------------------- | -------------------- |

| team\_id                          | 团队编号                 |

| company\_id                       | 公司编号                 |

| A\_sequence                       | A 处理顺序               |

| B\_sequence                       | B 接收顺序               |

| company\_enter\_pool\_time          | 公司进入锁定池时间            |

| locked\_pool\_size\_at\_B\_request    | B 请求下一家公司时锁定池大小      |

| eligible\_locked\_company\_ids      | 可抽取的已锁定公司集合          |

| chosen\_company\_id                | 被推送给 B 的公司           |

| company\_draw\_seed                | 公司池抽取随机种子            |

| frontier\_preassignment\_flag      | 是否使用 B\_PreA 预启动      |

| A\_active\_company\_id\_at\_B\_request | B 请求下一家公司时 A 正在处理的公司 |

| B\_empty\_pool\_duration            | 真实空窗时长               |

\## 十、迟到、no-show、退出与作废规则



\### 1. no-show



预报名但未到者不生成正式 participant\_id，不进入匹配池，不占用处理 slot。



记录：



| 字段                   | 含义    |

| -------------------- | ----- |

| registered\_id        | 预报名编号 |

| no\_show\_flag         | 是否未到  |

| scheduled\_session\_id | 预约场次  |



\### 2. 迟到



若迟到者到达时当前启动批次尚未完成匹配，可以进入当前匹配池。若当前批次已开始测试轮或正式实验，则不得插入已形成团队。



可选处理：



\* 进入下一启动批次；

\* 作为替补等待；

\* 改约下一场；

\* 支付到场补偿后离场。



\### 3. 团队正式开始前作废



若团队已形成但测试轮未通过、成员退出或出现设备故障，且尚未进入正式实验，可以作废该 team\_id。



处理规则：



1\. 记录 `team\_void\_before\_formal = 1`；

2\. 记录作废原因；

3\. 原 team\_id 不进入正式样本；

4\. 是否回收 treatment slot 需要系统统一规则，建议作废后该 slot 不进入正式样本审计，重新组成的新团队领取新的 slot。



\### 4. 正式实验开始后退出



若正式实验已经开始，一名成员退出或长时间失联，不替换队友。该团队继续按异常样本处理，并记录：



| 字段                     | 含义     |

| ---------------------- | ------ |

| partner\_dropout        | 是否队友退出 |

| dropout\_time           | 退出时间   |

| dropout\_role           | A 或 B  |

| severe\_disconnect\_flag | 是否严重断线 |

| affected\_company\_id    | 受影响公司  |

\## 十二、开发者应避免的错误实现



\### 1. 不要在预报名阶段分配处理组



错误做法：



```text

预报名时就给个人分配 early\_upgrade / batch / coop\_narrative

```



正确做法：



```text

实际到场并登录 → 随机匹配团队 → 分配 A/B → team\_id 领取 treatment slot

```



\### 2. 不要按登录顺序简单两两配对



错误做法：



```text

第 1 个登录 + 第 2 个登录 = team\_1

第 3 个登录 + 第 4 个登录 = team\_2

```



推荐做法：



```text

同一启动批次内随机打乱参与者，再两两配对

```



\### 3. 不要让先登录者固定为 A



A/B 角色必须由服务器随机分配。



\### 4. 不要让实验员选择处理组



处理 slot 必须隐藏，实验员和参与者均不可预测。



\### 5. 不要把材料 pattern 当成团队处理变量



`A\_macro\_B\_micro` 和 `A\_micro\_B\_macro` 是公司层面的材料属性，不是团队层面的实验组。



\### 6. 不要把公司推送随机误认为实验处理



公司池抽取、B\_PreA\_pending\_A 和真实空窗是统一流程状态，不是实验处理变量。

\## 十三、最终实施口径



本实验在实验室现场运行。预报名名单只用于排期，不参与正式随机化。参与者实际到场并完成登录后，进入本场次待匹配池。服务器在同一启动批次内随机匹配两名参与者组成团队，并在团队内部随机分配 A/B 角色。团队形成后，服务器从本场次隐藏随机 slot 序列中为该 team\_id 分配实验处理。



三个实验局均采用团队层面随机化。实验局 1 是被试内错位升级设计，所有团队从基础版 AI 开始，早升级团队第一次休息后升级，晚升级团队第二次休息后升级，所有团队最终都会升级；升级内容包括开放 AI 图片上传，以及 AI 模型由小参数模型升级为全参数模型。实验局 2 是团队层面的副线到达节奏实验，团队整局固定为 batch 或 continuous。实验局 3 是团队层面的合作叙事实验，团队整局固定为合作叙事组或中性信息组；合作叙事组内部，互补分工、验证留痕和共同责任三类主题在三个工作段中的展示顺序随机。实验局 3 的新闻流还需要在满足主题和组别约束后，随机生成 news\_id 展示顺序；同一团队内 A 和 B 使用相同新闻顺序。



宏观/微观材料分配属于公司层面的预设随机化。一些公司固定为 A\_macro\_B\_micro，另一些公司固定为 A\_micro\_B\_macro。某家公司一旦设定 pattern，在所有团队中保持一致。该 pattern 记录为 company\_id 的材料属性，不作为团队层面的实验处理变量。



公司推送机制是所有实验局共用的流程规则。首家公司 A/B 同步启动；后续公司优先从 A 已锁定公司池中随机推送给 B；若锁定池为空但 A 正在处理下一家公司，则 B 进入该公司的 B\_PreA\_pending\_A 状态；若二者都不存在，B 才进入真实空窗。所有匹配、角色分配、处理 slot、叙事主题顺序、新闻顺序、材料 pattern 和公司推送状态均必须日志化，以便实验后审计和分析。
