# 上线手册

> 状态：2026-06-05 当前项目上线准备手册
> 位置：`01_rules/上线手册.md`
> 用途：帮助后续协作者判断“离上线还差多少、先补什么、不要先补什么”

---

## 1. 先说结论

当前仓库**还没有到“一键上线”阶段**。

它现在的状态更准确地说是：

- 业务系统已经成形
- 本地开发链路已经能跑
- 服务器、域名、备案已经开始准备
- 但“可部署化”这一层还没有补完

因此，当前最合理的目标不是：

- 直接公网正式上线

而是分成三步：

1. 先把项目补成**可部署形态**
2. 再在服务器上做**裸 IP / 内网式部署验证**
3. 最后再接入**域名 / HTTPS / 正式实验**

---

## 2. 当前已经具备的上线基础

### 2.1 业务代码基础

当前仓库已经有：

- `apps/web`：Next.js 前端
- `apps/server`：NestJS 后端
- `apps/server/prisma`：PostgreSQL schema 与 migration
- admin、测试题、测试轮、正式轮、材料系统、副线系统、AI 区等核心业务代码

这意味着：

- 当前不是“从零开始写一个网站”
- 而是“把一个已经能本地运行的系统，补成能部署到服务器的系统”

### 2.2 本地运行基础

当前本地启动链路已经存在：

- 根脚本会先启动 PostgreSQL 容器
- 再分别起 server 和 web

当前关键文件：

- `package.json`
- `scripts/start-local.ps1`
- `infra/docker/docker-compose.yml`

### 2.3 基础服务器条件

当前已知准备：

- 已购买国内服务器
- 已购买域名
- ICP 备案正在进行

这意味着：

- 网络入口与合规入口已经开始推进
- 后续重点会落到“代码如何部署”和“服务器如何运行”

## 3. 当前离上线还差的核心工作

### 3.1 最大缺口：项目还没有正式部署包装

当前仓库里：

- 有 PostgreSQL 的 `docker-compose.yml`
- **没有** `apps/web/Dockerfile`
- **没有** `apps/server/Dockerfile`
- **没有** 项目级生产 `compose.yaml`
- **没有** Nginx / Caddy 的正式反向代理配置
- **没有** 成型的生产部署说明文档

所以当前并不是“只差买服务器和点部署按钮”。

当前真正缺的是：

- 把前端、后端、数据库、环境变量、反向代理整理成一套可部署结构

### 3.2 数据库上线运维还没收口

当前 Prisma schema 已经比较完整，但生产运维层面还缺：

- 生产库初始化流程
- migration 执行步骤
- 生产环境 seed / 初始配置导入方式
- 备份与恢复方案
- 数据口径继续补齐的优先级排序

这里要区分两类问题：

- **系统能不能上线运行**
- **实验变量记录是否已经到分析理想态**

当前并不是因为“变量还没记满”就完全不能上线；
但如果要进入**正式实验采数阶段**，数据库口径还要继续收紧。

### 3.3 文件与材料当前仍是单机本地存储思路

当前后端直接使用：

- `storage/materials`
- `storage/tmp`

并通过本机静态路径暴露材料。

这对第一版单机实验站来说**不是不能用**，但要清楚它的含义：

- 文件存储依赖这台服务器本地磁盘
- 换机迁移会更麻烦
- 需要自己做备份
- 还没有真正接入对象存储

因此：

- 这不是当前必须立刻解决的问题
- 但它是上线后必须心里有数的风险点

### 3.4 环境变量与生产配置还没整理成型

当前 `.env.example` 很初级，只覆盖了极少量字段。

真正上线至少要明确：

- `DATABASE_URL`
- server 端口
- web 到 server 的 API 地址
- AI Key / Base URL / Model
- 域名、CORS、生产访问地址
- 材料与上传路径相关配置

换句话说：

- 业务代码已有不少逻辑
- 但“生产配置的明确化和模板化”还没做完

## 4. 现在最容易走偏的点

### 4.1 不要先把注意力全压在“变量还不够多”

变量记录继续补是对的，但它不是现在唯一优先级。

如果部署结构没搭出来：

- 你会一直停在本地
- 也很难真正暴露出线上问题

因此当前优先级应理解为：

1. 先补部署结构
2. 再上线内测
3. 再继续补实验变量细口径

### 4.2 不要过早把精力用在高阶云产品上

当前不要优先陷进：

- COS / 对象存储接线
- CDN
- 多机部署
- 复杂监控栈
- 各种面板化平台

第一版上线更重要的是：

- 单机能稳定跑
- 数据库能连
- 前后端能起
- 域名能访问
- 备份能做

### 4.3 不要把“能上线”误解成“正式实验可放心跑”

这两件事不是同一个层级：

- **能上线**：服务器能跑、能访问、能提交数据
- **适合正式实验**：流程稳定、日志可靠、数据口径明确、能备份、能恢复、已压测

## 5. 建议的上线路线

### 阶段 1：先把项目补成“可部署形态”

目标：

- 在本机上用接近生产的方式，一条命令起完整服务

这一阶段优先补：

1. `apps/web/Dockerfile`
2. `apps/server/Dockerfile`
3. 根目录生产 `compose.yaml`
4. 生产环境变量模板
5. 最小部署说明文档

### 阶段 2：服务器上先做“裸 IP 部署验证”

目标：

- 暂时不先接域名
- 暂时不先接 HTTPS
- 先通过服务器 IP 验证：
  - web 能起
  - server 能起
  - DB 能连
  - 材料能读
  - 表单能提交

### 阶段 3：再接域名、备案、HTTPS

这一阶段处理：

- ICP 备案通过后的解析
- 反向代理
- HTTPS
- 正式公网入口

### 阶段 4：最后做正式实验级收口

这一阶段处理：

- 备份
- 恢复
- 日志
- 健康检查
- 压测
- 正式彩排
- 数据记录口径查漏补缺

## 7. 需要先掌握的最小知识树

当前不要泛学 Linux。优先只学能支撑本项目上线的最小集。

### 7.1 进程

要知道：

- 什么是一个运行中的服务
- web、server、postgres 分别是独立进程 / 容器
- 如何判断它是不是活着

### 7.2 端口

要知道：

- `3000`、`3001`、`5432` 分别是什么
- 浏览器为什么不能直接无脑访问所有端口
- 为什么最后通常只暴露 `80/443`

### 7.3 环境变量

要知道：

- 为什么本地和线上配置不同
- 为什么不能把数据库地址、AI Key 写死在代码里

### 7.4 反向代理

要知道：

- 为什么用户访问的是域名，而不是 `:3000` / `:3001`
- 为什么需要 Nginx / Caddy 把外部请求转发给 web / server

### 7.5 持久化

要知道：

- 数据库数据和材料文件为什么不能跟容器一起随便删
- 哪些目录必须保留

### 7.6 数据库迁移

要知道：

- schema 改了以后，线上数据库不会自己变
- 必须有明确的 migration 执行流程

## 8. 当前最推荐的部署思路

对这个项目，当前更推荐的长期方案是：

- **Docker 化**
- 单机部署
- 后续再考虑部署面板或更自动化方案

### 为什么不是现在直接走“真一键”

像下面这些路线：

- Coolify
- Dokploy
- CapRover

本质上都要求你先把应用补成：

- 有 Dockerfile
- 有 compose
- 有清楚的环境变量

所以它们不是“替你省掉基础工作”，而是“在你把基础工作做完后，再让部署更省力”。

因此当前阶段不应理解为：

- “去找一个平台一键上线”

而应理解为：

- “先把项目补成任何平台都能接得住的可部署结构”

---

## 9. 当前状态的一句话判断

当前项目已经不是“离上线非常遥远”的阶段。

但它也还不是：

- 服务器一买
- 域名一配
- 直接就能公网稳定运行

更准确的判断是：

- **业务系统已成形**
- **部署结构未成形**

当前真正要补的不是“再想一个功能”，而是：

- 把现有系统收口成一套可部署、可迁移、可排障的上线结构

---

## 10. 当前默认行动顺序

如果后续继续推进上线，默认按这个顺序做：

1. 先补 Dockerfile
2. 再补生产 compose
3. 再补生产环境变量模板
4. 再在本机验证“生产式启动”
5. 再搬到服务器裸 IP 跑通
6. 再接域名与 HTTPS
7. 最后做正式实验前的备份 / 压测 / 彩排

这条顺序优先级最高，不要轻易倒过来。
