# 时间戳变量保存方案

## 0. 背景

本方案解决一类新的论文分析变量：被试在主线、旁线、副线任务和 AI 等待期间如何分配注意力，以及从一次任务切换或 AI 返回之后多久恢复主线操作。

这些变量不能只靠最终答案、AI 聊天次数或副线正确率推断，必须保存可回溯的时间戳。当前系统已经有 `events/events.jsonl`、`side_events.jsonl`、`ai_chat.jsonl`、`company_metadata.json` 等原始文件，本方案不推翻这些结构，而是在被试目录下新增一个分析友好的汇总文件：

```text
sessions/SESSION_CODE/participants/PARTICIPANT_ID/timestamps.json
```

`events.jsonl` 仍然是审计真相源；`timestamps.json` 是从原始事件、AI 日志、副线事件和公司任务状态整理出来的被试级时间戳汇总与派生变量文件。

## 1. 需要保存多少类时间戳

师兄变量表里的 A/B/C/D/E 是 5 个核心锚点，但为了稳定计算分母、跨公司汇总、AI 等待期间行为和恢复时滞，实际需要保存 11 类时间戳事件。

| 编号 | 事件名 | 含义 | 主要用途 |
|---|---|---|---|
| T1 | `work_segment_started` | 正式工作段开始 | 计算工作段分母、裁剪主副线时间 |
| T2 | `work_segment_ended` | 正式工作段结束 | 计算工作段分母、裁剪未闭合区间 |
| T3 | `company_context_entered` | 被试进入或获得某家公司任务上下文 | 按公司归属主线时间、AI 等待窗 |
| T4 | `company_context_exited` | 公司任务结束、提交、切换或段末冻结 | 关闭公司级时间窗 |
| T5 | `side_area_entered` | 从主线进入副线区域，等同编号 A | 计算副线时间、切换次数、AI 等待期间是否进副线 |
| T6 | `main_area_returned` | 从副线返回主界面，等同编号 B | 计算副线区间结束、恢复时滞起点 |
| T7 | `mainline_activity` | 主线有意义操作，作为编号 C 的原始来源 | 计算 `RestartDelay` 和 `ReturnDelayAfterAIResponse` |
| T8 | `ai_wait_started` | 点击发送 AI，请求开始，等同编号 D | 定义 AI 等待期起点 |
| T9 | `ai_wait_ended` | AI 流式输出结束或失败结束，等同编号 E | 定义 AI 等待期终点 |
| T10 | `side_activity` | 副线内打开、滚动、作答、使用副线 AI 等动作 | 辅助判断 AI 等待期内副线活动 |
| T11 | `main_context_activity` | 主线材料查看、表单聚焦、表单编辑、主线 AI 以外动作 | 辅助判断 AI 等待期内主线活动 |

实现时不建议把所有键盘输入都逐字保存成事件。更稳健的做法是：前端维护“待恢复锚点”，在副线返回或 AI 返回之后，只记录下一次有意义主线操作为 `mainline_activity`。有意义主线操作优先使用表单内容发生变化的 `edit`，其次可记录材料点击、主线表单聚焦等 `focus` / `material_view`，并在事件里保存 `activityKind`。

## 2. A/B/C/D/E 锚点定义

| 锚点 | 保存事件 | 解释 |
|---|---|---|
| A | `side_area_entered.at` | 被试进入副线区。点击副线任务、点击副线入口、滚动/聚焦到副线面板，只要前端状态进入副线，都记 A。 |
| B | `main_area_returned.at` | 被试点击返回主界面或前端状态从副线回到主线。 |
| C | `mainline_activity.at` | B 或 E 之后第一次主线有意义操作。默认用第一次表单内容变化；若研究口径希望更宽，可用第一次主线表单聚焦或材料点击。 |
| D | `ai_wait_started.at` | AI 点击发送成功进入等待，按 `requestId` 绑定。 |
| E | `ai_wait_ended.at` | AI assistant 消息流式输出结束、失败结束或超时结束。 |

C 不需要保存成“每一次输入时间戳”。推荐逻辑是一次锚点只产生一次 C：

1. 发生 B 后，前端设置 `pendingResumeAnchor = side_return`。
2. 后续第一次主线表单编辑、主线材料点击或主线表单聚焦触发 `mainline_activity`。
3. 该事件带上 `anchorType = side_return` 和 `anchorEventId`，后续动作不再为同一锚点重复生成 C。
4. 发生 E 后同理设置 `pendingResumeAnchor = ai_response_end`，下一次主线有意义操作用于计算 AI 返回后的恢复时滞。

## 3. 派生变量计算逻辑

### 3.1 MainTimeShare / SideTimeShare

时间分母默认是被试在正式工作段内的有效工作时间：

```text
eligibleWorkMs = sum(work_segment_ended - work_segment_started)
```

副线时间是所有 `side_area_entered -> main_area_returned` 区间与正式工作段区间取交集后的总时长：

```text
sideMs = sum(overlap(sideInterval, formalWorkSegments))
mainMs = max(eligibleWorkMs - sideMs, 0)
SideTimeShare = sideMs / eligibleWorkMs
MainTimeShare = mainMs / eligibleWorkMs
```

如果要按公司计算，则再把区间与 `company_context_entered -> company_context_exited` 取交集。对投资经理可能存在空窗，公司级分母只统计有公司上下文的时间；被试级总分母仍保留整个正式工作段，便于识别空窗和等待。

### 3.2 SwitchCount

默认论文变量 `SwitchCount` 统计主线进入副线的次数，也就是 `side_area_entered` 次数。为了后续可解释，`timestamps.json` 同时保存：

| 字段 | 含义 |
|---|---|
| `switchVisitCount` | 副线访问次数，等于 A 次数 |
| `switchTransitionCount` | 主副线状态转换次数，A 和 B 都计数 |
| `unclosedSideVisitCount` | 有 A 但没有 B、需要用段末裁剪的副线访问次数 |

### 3.3 RestartDelay

`RestartDelay` 是从副线返回主线后的恢复时滞：

```text
RestartDelay = first(C after B) - B
```

计算时每个 `main_area_returned` 生成一个候选恢复窗，寻找其后的第一个 `mainline_activity`。默认只接受 `activityKind = edit` 作为严格恢复；同时保留宽口径 `activityKind in [edit, focus, material_view]` 的备选值。

`timestamps.json` 中建议保存：

```json
{
  "afterSideReturn": [
    {
      "returnEventId": "evt_side_return_001",
      "returnedAt": "2026-06-17T14:03:20.000+08:00",
      "firstMainActivityAt": "2026-06-17T14:03:31.000+08:00",
      "activityKind": "edit",
      "delayMs": 11000
    }
  ],
  "meanRestartDelayMs": 11000,
  "medianRestartDelayMs": 11000
}
```

### 3.4 AI 等待期间行为

每次 AI 请求形成一个等待窗：

```text
aiWaitWindow = [D, E]
```

当前实现口径里，E 是服务端写入的 `ai_wait_ended.at`，不是前端看到第一个 token、也不是浏览器渲染完成时间。流式成功时，后端在上游流读取完毕、整理最终 assistant 文本并完成本次 AI 响应收口后写入 E；非流式、mock/provider 降级路径同理在服务端响应收口后写入 E；失败或超时时，在后端捕获 provider/API 错误并结束本次请求后写入 E。E 使用同一个 `requestId` 与 D 绑定，并与 `AiMessageLog.completedAt` / `latencyMs` 互相校验。若服务器进程在 D 之后崩溃、网络中断导致后端没有走到收口逻辑，则 E 可能缺失，导出时必须标记 `missing_ai_wait_end`，不能伪造结束时间。

对每个等待窗分别判断：

| 变量 | 判断逻辑 |
|---|---|
| `SideDuringAIWait` | 等待窗内存在副线区间交集，或存在 `side_activity` |
| `MainDuringAIWait` | 等待窗内存在主线表单编辑、材料查看、主线表单聚焦等 `main_context_activity` |
| `IdleDuringAIWait` | 等待窗内没有副线活动，也没有主线活动 |

这三个变量不是互斥的前两个。被试可能在一次 AI 等待期间先继续主线、再进入副线，此时 `SideDuringAIWait = true` 且 `MainDuringAIWait = true`，`IdleDuringAIWait = false`。只有完全没有可识别操作时，`IdleDuringAIWait = true`。

### 3.5 ReturnDelayAfterAIResponse

AI 返回后恢复主线操作的时滞：

```text
ReturnDelayAfterAIResponse = first(C after E) - E
```

这里的 C 和 `RestartDelay` 共用 `mainline_activity` 事件，但 `anchorType = ai_response_end`。如果 AI 返回后被试先进入副线，再回主线，则保存原始序列，并在派生字段中标记：

```json
{
  "requestId": "ai_001",
  "responseEndedAt": "2026-06-17T14:04:05.000+08:00",
  "firstMainActivityAt": "2026-06-17T14:04:40.000+08:00",
  "delayMs": 35000,
  "sideBeforeReturn": true
}
```

### 3.6 TotalMainTime / TotalSideTime

个人总量：

```text
TotalSideTime = sideMs
TotalMainTime = mainMs
```

团队总量可以在 session 后处理阶段把同一 session 两名 participant 的值相加：

```text
TeamTotalSideTime = participantA.sideMs + participantB.sideMs
TeamTotalMainTime = participantA.mainMs + participantB.mainMs
```

团队层不建议直接写进 participant 的 `timestamps.json`，可以后续由 session 汇总脚本生成，避免 A/B 文件互相依赖。

## 4. `timestamps.json` 放置位置

建议放在 participant 根目录：

```text
sessions/SESSION_CODE/participants/PARTICIPANT_ID/timestamps.json
```

理由：

1. 时间戳变量跨公司、跨副线、跨 AI 请求，不适合塞进单家公司目录。
2. 它是单个被试的完整行为时间线，和 `variables.json` 同级最容易读取。
3. session 层只保存共享事实，不应保存个人行为轨迹。
4. `events/events.jsonl` 保留原始事件；`timestamps.json` 保存清洗后的区间、锚点和派生变量。

## 6. 原始事件当前实现

当前已通过 `POST /experiment/session/:code/timestamps/event` 和 `AiService` 在 `ExperimentEvent` 中写入以下时间戳事件。导出时原始事件继续进入 `events/events.jsonl`，再由 `ExportService` 整理成 `timestamps.json`。

| eventType | payload 关键字段 |
|---|---|
| `work_segment_started` | `workSegment`, `segmentIndex` |
| `work_segment_ended` | `workSegment`, `segmentIndex`, `endedReason` |
| `company_context_entered` | `companyId`, `taskAssignmentId`, `workSegment` |
| `company_context_exited` | `companyId`, `taskAssignmentId`, `reason` |
| `side_area_entered` | `companyId`, `taskAssignmentId`, `source`, `sideTaskPlanId` |
| `main_area_returned` | `companyId`, `taskAssignmentId`, `source` |
| `mainline_activity` | `companyId`, `taskAssignmentId`, `activityKind`, `anchorType`, `anchorEventId` |
| `main_context_activity` | `companyId`, `taskAssignmentId`, `activityKind` |
| `side_activity` | `sideTaskPlanId`, `activityKind` |
| `ai_wait_started` | `requestId`, `contextType`, `companyId`, `sideTaskPlanId`, `aiLevel` |
| `ai_wait_ended` | `requestId`, `contextType`, `endedBy`, `latencyMs`, `providerStatus` |

AI 的 D/E 也可以从 `AiMessageLog.createdAt/completedAt/latencyMs` 回推，但显式保存 `ai_wait_started` 和 `ai_wait_ended` 更便于处理流式输出、失败、超时和断线恢复。

## 7. 计算步骤

1. 读取 `session_metadata.json` 和各 `segment_metadata.json`，得到正式工作段区间。
2. 读取公司目录下 `company_metadata.json`，得到每家公司上下文、提交、解锁、完成时间。
3. 读取 `events/events.jsonl`，抽取主副线切换、主线活动、AI 等待开始/结束等事件。
4. 读取 `side_tasks/side_events.jsonl`，补充副线打开、作答、提醒等副线活动。
5. 读取主线和副线 `ai_chat.jsonl`，校验 `ai_wait_started/ended` 是否与 AI 消息一致。
6. 按时间排序，构造正式工作段、公司、side visit、AI wait 四类区间。
7. 将 side visit 与正式工作段、公司上下文取交集，计算主线/副线时间。
8. 对每个 B 找后续第一个 C，计算 `RestartDelay`。
9. 对每个 AI wait window 找区间内主线/副线活动，计算 `SideDuringAIWait`、`MainDuringAIWait`、`IdleDuringAIWait`。
10. 对每个 E 找后续第一个 C，计算 `ReturnDelayAfterAIResponse`。
11. 汇总到 `derived`，并生成质量标记。

## 8. 质量标记

当前 `timestamps.json.qualityFlags` 已覆盖以下情况：

| flag | 含义 |
|---|---|
| `unclosed_side_interval` | 有 A 无 B，用段末或导出时间裁剪 |
| `missing_work_segment_boundary` | 缺少工作段开始或结束 |
| `missing_ai_wait_end` | 有 D 无 E，AI 等待窗只能按失败或导出时间关闭 |
| `negative_or_zero_interval` | 事件顺序异常，出现负时长或零时长 |
| `main_activity_missing_after_side_return` | B 后直到段末都没有 C |
| `main_activity_missing_after_ai_response` | E 后直到段末都没有 C |
| `event_clock_out_of_order` | 同一 participant 事件时间明显乱序；当前作为后续扩展口径保留 |

## 9. 与现有导出文件关系

| 现有文件 | 和时间戳模块的关系 |
|---|---|
| `variables.json` | 继续保存核心摘要，当前已增加 `timing` 摘要字段，但不放完整时间线 |
| `timestamps.json` | 当前已新增，被试级时间戳汇总和派生变量主文件 |
| `events/events.jsonl` | 原始审计源，不能被 `timestamps.json` 替代 |
| `side_tasks/side_events.jsonl` | 副线原始事件源 |
| `ai_chat.jsonl` | AI 请求和响应时间的校验源 |
| `company_metadata.json` | 公司上下文、A/B 提交和解锁时间源 |
