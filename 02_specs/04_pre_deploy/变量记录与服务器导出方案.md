# 变量记录与服务器导出方案

## 1. 文档目标

本方案定义正式实验运行时需要记录哪些数据、如何落库、如何组织服务器侧导出文件，以及后续上线时如何保留扩展空间。

核心目标不是马上把所有论文变量都算完，而是先保证：

1. 实验操纵变量一定能算出来。
2. 每个被试在每家公司、每个副线任务、每次 AI 聊天中的原始数据可以回溯。
3. 数据结构足够稳定，后续上线、备份、清洗和论文分析都能接上。
4. 大体量数据不通过 admin 页面一次性直接返回，而是在服务器生成导出归档。

## 2. 当前口径

本项目是实验平台，不是普通业务系统。记录层要分清三类数据：

### 2.1 事件数据

记录“什么时候做了什么”。

典型例子：

- 登录、成组、角色随机完成
- 指导语曝光、准备页点击
- 测试题提交、测试轮开始、正式轮开始
- 尽调员任务开始、自动提交、冻结、恢复
- 投资经理公司分配、查看尽调员信息、提交决策、提交反馈
- 副线任务释放、展示、回答、归档
- AI 请求、AI 回复、图片上传
- 问卷曝光、问卷提交
- 异常事件：断线、刷新、重复提交、超时、材料加载失败、AI 请求失败

### 2.2 内容数据

记录“当时写了什么 / 看到什么 / 交了什么”。

典型例子：

- 尽调员每家公司答题区完整内容
- 投资经理每家公司决策区完整内容
- 投资经理每家公司反馈区完整内容
- 草稿保存快照
- 5 分钟解锁瞬间快照
- 工作段结束冻结快照
- 恢复来源快照
- 副线任务题干、选项、被试答案、正确答案、是否正确
- 问卷题目、选项、被试答案

### 2.3 AI 聊天与上下文数据

记录“AI 当时在什么任务上下文里发生聊天”。

这里不做复杂 prompt 组装快照，不把材料全文反复塞进 AI 上下文表。当前只要求能导出聊天记录，并附带足够的任务上下文索引。

每条 AI 聊天至少需要保留：

- `sessionId`
- `participantId`
- `role`
- `contextType`: `main` / `side`
- `companyId`
- `taskAssignmentId`，如果当次聊天属于某家公司
- `sideTaskPlanId`，如果当次聊天属于副线任务
- `phase`
- `segmentIndex`
- `aiLevel`
- `modelVersion`
- `imageUploadEnabled`
- `messageRole`: `user` / `assistant`
- `requestId`
- `content`
- `attachments`
- `createdAt`

额外任务上下文建议通过导出时 join 出来：

- 公司名称、公司顺序、材料版本或材料元数据
- 角色可见材料目录
- A 信息是否已解锁
- B 是否已点击查看尽调员信息
- A 处理该公司时的 AI 状态
- B pre-A / post-A 阶段 AI 状态
- 副线题干、选项、主题、叙事组、推送模式

## 3. 数据组织原则

### 3.1 运行表和研究记录表分离

现有业务表继续服务页面运行：

- `Session`
- `Pairing`
- `TaskAssignment`
- `TaskSnapshot`
- `QuestionnaireResponse`
- `AiMessageLog`
- `SideTaskPlan`
- `SideTaskExposureLog`
- `SideTaskResponseLog`
- `RandomizationAudit`

新增实验审计层负责稳定研究记录：

- `SessionConfigSnapshot`
- `BehaviorEvent`

### 3.2 大数据不走 admin 直接响应

admin 页面可以触发导出任务，但不应该直接返回完整大 JSON。

推荐方式：

1. admin 点击“生成实验数据归档”。
2. 后端创建 `ExportJob`。
3. 后端在服务器本地生成目录和压缩包。
4. admin 页面轮询任务状态。
5. 生成完成后，admin 下载 zip。

本地开发阶段可先同步生成，正式上线前再改成后台任务队列。

### 3.3 每个被试一套目录

导出包的主结构按 session / participant 分层。每个被试目录下再分三类子文件夹：

```text
exports/
  experiment-export-20260605-153000/
    manifest.json
    sessions/
      SESSION_CODE/
        session.json
        randomization.json
        participants/
          PARTICIPANT_ID/
            participant.json
            events/
              events.jsonl
            content/
              companies/
                COMPANY_SEQUENCE_001_COMPANY_ID/
                  mainline.json
                  snapshots.jsonl
                COMPANY_SEQUENCE_002_COMPANY_ID/
                  mainline.json
                  snapshots.jsonl
              side_tasks/
                side_tasks.jsonl
              questionnaires/
                questionnaires.jsonl
            ai/
              main_chat.jsonl
              side_chat.jsonl
```

说明：

- `events/` 放事件数据。
- `content/` 放答题、提交、副线、问卷、快照等内容数据。
- `ai/` 放 AI 聊天记录和轻量上下文索引。
- 大数组优先用 `jsonl`，方便流式写入和后续 Python/R 读取。
- 小型摘要文件用 `json`，方便人工查看。

## 4. 建议新增数据库表

### 4.1 SessionConfigSnapshot

保存 session 创建时的完整配置快照，避免 admin 后续改配置影响旧 session 的解释。

建议字段：

```text
id                    String
sessionId             String @unique
schemaVersion          Int
experimentMode         String
experimentSnapshot     Json
durationConfig         Json
aiConfig               Json
sideTaskConfig         Json
narrativeConfig        Json
questionnaireConfig    Json
instructionConfig      Json
materialConfig         Json?
createdAt              DateTime
```

其中：

- `experimentSnapshot` 保存实验模式、随机化结果、固定变量、seed。
- `durationConfig` 保存练习轮、正式工作段、休息段、A 单家公司 5 分钟规则。
- `aiConfig` 保存基础版/升级版模型名、分段 AI 状态、图片能力。
- `sideTaskConfig` 保存推送模式、提醒频率、题目计划版本。
- `narrativeConfig` 保存叙事组、主题顺序。
- `questionnaireConfig` 保存测试题模板、休息页问卷模板版本。
- `instructionConfig` 保存指导语积木版本和当时文本。
- `materialConfig` 预留材料批次、材料版本、对象存储 key 等上线信息。

### 4.2 BehaviorEvent

统一记录关键行为和系统事件。

建议字段：

```text
id                    String
sessionId             String
participantId          String?
taskAssignmentId       String?
companyId              String?
sideTaskPlanId         String?
role                   String?
eventType              String
phase                  String?
segmentIndex           Int?
serverTime             DateTime
clientTime             DateTime?
payload                Json?
schemaVersion          Int
createdAt              DateTime
```

建议索引：

```text
@@index([sessionId, participantId])
@@index([sessionId, eventType])
@@index([taskAssignmentId])
@@index([companyId])
@@index([sideTaskPlanId])
@@index([serverTime])
```

## 5. 现有表需要补强的字段

### 5.1 AiMessageLog

当前已经记录 AI 消息本体。建议补充轻量任务索引：

```text
taskAssignmentId String?
sideTaskPlanId   String?
providerStatus   String?
latencyMs        Int?
errorMessage     String?
```

不建议现在新增复杂 `AiContextSnapshot` 表。AI 上下文导出以聊天记录为主，任务上下文通过导出时关联现有表补齐。

### 5.2 TaskAssignment

当前已保存：

- `aDraft`
- `bDraft`
- `bFeedbackDraft`
- `aStartedAt`
- `aDeadlineAt`
- `aSubmittedAt`
- `aUnlockedForBAt`
- `bViewedAInfoAt`
- `bCompletedAt`
- `aAiLevelAtWindow`
- `bPreAAiLevel`
- `bPostAAiLevel`
- `crossUpgradeBoundaryFlag`

建议确认导出时把这些字段完整保留，不要只导出摘要。

如后续需要区分“草稿”和“最终提交”更清楚，可以新增：

```text
aFinalSubmission       Json?
bFinalDecision         Json?
bFinalFeedback         Json?
```

但最小改动方案可以先复用当前 draft 字段，并通过提交事件时间解释其最终状态。

### 5.3 SideTaskExposureLog

当前可以记录副线释放、展示、回答等事件。建议统一 payload 结构：

```json
{
  "answer": "A",
  "isCorrect": true,
  "clientTime": "...",
  "reactionTimeMs": 12345,
  "displayDurationMs": 8000
}
```

## 9. 服务器侧导出接口建议

### 9.1 创建导出任务

```text
POST /admin/export-jobs
```

请求：

```json
{
  "scope": {
    "sessionIds": [],
    "includeIncompleteSessions": true
  },
  "format": "participant_folder_zip"
}
```

返回：

```json
{
  "ok": true,
  "jobId": "...",
  "status": "pending"
}
```

### 9.2 查询导出任务

```text
GET /admin/export-jobs/:jobId
```

返回：

```json
{
  "ok": true,
  "job": {
    "id": "...",
    "status": "completed",
    "progress": 100,
    "archivePath": "storage/exports/experiment-export-..."
  }
}
```

### 9.3 下载导出包

```text
GET /admin/export-jobs/:jobId/download
```

本地开发阶段可直接从 `storage/exports` 读取 zip。上线后建议改为对象存储或受控临时 URL。

## 11. 和上线部署的关系

本方案和上线部署有三处衔接：

1. 导出文件不要永久依赖本地开发机路径。当前可以先写入 `storage/exports`，上线后改为对象存储。
2. 导出包应保存 `manifest.json` 和 schemaVersion，避免后续字段升级后旧数据无法解释。
3. 大文件下载需要受控访问。上线后不建议公开静态暴露，应走 admin 鉴权接口或临时签名 URL。

## 12. 最小实现顺序

### 第一阶段：文档与 schema

1. 新增本方案文档。
2. Prisma 新增 `BehaviorEvent`、`SessionConfigSnapshot`、`ExportJob`。
3. `AiMessageLog` 补 `taskAssignmentId`、`sideTaskPlanId`、`providerStatus`、`latencyMs`、`errorMessage`。
4. 生成 migration 并跑 `prisma generate`。

### 第二阶段：审计写入

1. 新增 `experiment-audit` 服务。
2. session 初始化时写 `SessionConfigSnapshot`。
3. 用 `BehaviorEvent` 覆盖关键事件：登录、成组、ready、测试题、A 自动提交、B 查看 A、B 提交、副线、问卷、AI。
4. 保留现有 `TaskProgress`，但不再把它当正式研究记录层。

### 第三阶段：内容完整性

1. 确认 `TaskAssignment` 的三块答题内容完整导出。
2. 确认 `TaskSnapshot` 里的关键快照完整导出。
3. 副线任务导出时 join `SideTaskItem`、`SideTaskPlan`、`SideTaskExposureLog`。
4. 问卷导出时 join `QuestionnaireTemplate`。

### 第四阶段：服务器导出

1. 新增 `export-jobs` 接口。
2. 生成目录结构。
3. 按被试写 `participant.json`。
4. 按公司写 `mainline.json` 和 `snapshots.jsonl`。
5. 写 `events.jsonl`、`main_chat.jsonl`、`side_chat.jsonl`、`side_tasks.jsonl`、`questionnaires.jsonl`。
6. 写 `manifest.json`。
7. 压缩为 zip。
8. admin 页面增加生成、查看状态、下载入口。

### 第五阶段：验证

1. 构造一组完整双人 session。
2. 覆盖实验 1/2/3 三种模式。
3. 每种模式至少跑一家公司、一次副线、一次 AI、一次问卷。
4. 导出 zip 并人工检查目录结构。
5. 用脚本读取 json/jsonl，验证核心变量能算出来。

## 13. 必须能算出的变量

导出后至少要能直接计算：

- 实验模式
- AI 升级组别
- 副线提醒模式
- 叙事组别
- 主题顺序
- 三段 AI 状态
- 角色分配结果
- 公司顺序
- B 公司分配方式
- 每家公司 A 处理时 AI 状态
- 每家公司 B pre-A / post-A AI 状态
- 是否跨升级边界
- A 单家公司处理时长
- A 是否自动提交
- B 是否查看尽调信息
- B 查看尽调信息延迟
- B 提交延迟
- AI 请求次数
- AI 基础版/升级版请求次数
- 图片上传次数
- 副线释放数
- 副线回答数
- 副线正确率
- 副线反应时
- 问卷答案与提交时间

## 14. 当前不做

当前阶段不做：

- 不做复杂 AI prompt 还原系统。
- 不把每次 AI 请求的全部材料全文复制进日志。
- 不在 admin 接口直接返回完整超大 JSON。
- 不引入复杂数据仓库。
- 不先做论文变量计算 UI。

当前优先目标是：原始数据完整、结构稳定、服务器可导出、后续可计算。
