# 数据库文件夹手册

## 0. 这份手册怎么用

这份手册只解决一件事：拿到 admin 导出的实验数据包后，最快速度看懂每个文件夹、每个 JSON / JSONL 文件、每类指标分别代表什么。

当前导出包不是传统“一个数据库表格文件”，而是把数据库按实验逻辑整理成文件夹树：

```text
export
  -> sessions
    -> 每个 session
      -> 每个 participant
        -> 元数据 / 问卷 / 练习轮 / 正式段 / 副线 / 事件 / 图片附件
```

阅读顺序建议：

1. 先看 `manifest.json`，确认导出范围和总数量。
2. 再进 `sessions/SESSION_CODE/`，看这个 session 的共享事实。
3. 再进 `participants/PARTICIPANT_ID/`，看单个被试的全部数据。
4. 如果只想快速算变量，先看 `variables.json`。
5. 如果要追原始过程，再看 `events.jsonl`、`answer_content.json`、`ai_chat.jsonl`、`side_responses.jsonl`。

时间字段默认是 ISO 字符串。JSONL 表示“一行一个 JSON 对象”，适合大量事件、AI 聊天和副线题。

## 1. 导出包总结构

当前实际导出包主干：

```text
EXPORT_ROOT/
  manifest.json
  variable_implementation_checklist.jsonl
  variable_implementation_checklist.md
  variable_implementation_checklist_meta.json
  variable_implementation_summary.json
  sessions/
    SESSION_CODE/
      session_metadata.json
      randomization.json
      participants/
        PARTICIPANT_ID/
          participant_metadata.json
          variables.json
          questionnaires/
            practice_quiz.json
            break_1.json
            break_2.json
          practice_round/
            round_metadata.json
            companies/
              COMPANY_CODE__COMPANY_ID/
                company_metadata.json
                answer_content.json
                snapshots.jsonl
                ai_chat.jsonl
            side_tasks.jsonl
            side_ai_chat.jsonl
          formal_segments/
            segment_1/
              segment_metadata.json
              companies/
                COMPANY_CODE__COMPANY_ID/
                  company_metadata.json
                  answer_content.json
                  snapshots.jsonl
                  ai_chat.jsonl
            segment_2/
              segment_metadata.json
              companies/
                COMPANY_CODE__COMPANY_ID/
                  company_metadata.json
                  answer_content.json
                  snapshots.jsonl
                  ai_chat.jsonl
            segment_3/
              segment_metadata.json
              companies/
                COMPANY_CODE__COMPANY_ID/
                  company_metadata.json
                  answer_content.json
                  snapshots.jsonl
                  ai_chat.jsonl
          side_tasks/
            side_plan.json
            side_responses.jsonl
            side_events.jsonl
            side_ai_chat.jsonl
          events/
            events.jsonl
          attachments/
            images/
              IMAGE_FILE.png
```

重要口径：

- `session` 保存共享事实：配置快照、随机化、公司顺序、B 动态分配日志。
- `participant` 保存个人事实：问卷、答题、AI、事件、副线作答、图片附件。
- A/B 的副线计划可以相同，但作答、打开、提醒、AI 必须分 participant 保存。
- 公司目录名是 `companyCode__companyId`，用于避免多个公司代号重复时互相覆盖。
- 图片不塞进 JSON；JSON 只保存 `attachments/images/...` 相对路径。

## 2. 根目录文件

### 2.1 manifest.json

用途：看这个导出包是什么时候生成的、导出了多少数据。

主要字段：

| 字段 | 含义 | 来源 |
|---|---|---|
| `schemaVersion` | 导出结构版本 | 导出服务 |
| `exportId` | 本次导出 ID | `ExportJob` + 时间戳 |
| `exportedAt` | 导出时间 | `ExportJob.startedAt` |
| `appVersion` | 应用版本，目前本地为 `local-dev` | 导出服务 |
| `scope.sessionCodes` | 指定导出的 session code；空数组表示全部 | admin 导出参数 |
| `scope.includeIncompleteSessions` | 是否包含未完成 session | admin 导出参数 |
| `counts.sessions` | session 数量 | `Session` |
| `counts.participants` | participant 数量 | `Pairing` / `Participant` |
| `counts.companies` | 公司库公司数量 | `Company` |
| `counts.sideTaskPlans` | 副线计划条数 | `SideTaskPlan` |
| `counts.aiMessages` | AI 消息行数 | `AiMessageLog` |
| `counts.imageAttachments` | 带图片附件的 AI 消息数 | `AiMessageLog.attachments` |

### 2.2 variable_implementation_summary.json

用途：本次导出包的自动体检摘要。

主要字段：

| 字段 | 含义 |
|---|---|
| `checkedAt` | 自检执行时间 |
| `rawSaved` | 已通过的原始保存项 |
| `risks` | 有风险但不一定缺数据的项，例如旧 session 没有新事件 |
| `missingSource` | 确认缺少原始来源的项 |
| `postProcessable` | 不实时存储、但可由原始数据后处理的变量 |
| `outOfPhaseOne` | 第一版明确不做的后续模块 |

### 2.3 variable_implementation_checklist.jsonl / .md

用途：逐项体检表。

字段：

| 字段 | 含义 |
|---|---|
| `variable` | 被检查的记录项 |
| `source` | 保存来源表或文件 |
| `trigger` | 写入触发点 |
| `exportLocation` | 导出位置 |
| `status` | `已通过` / `有风险` / `缺失` / `可后处理` / `后续模块` |
| `notes` | 备注与风险 |

## 3. Session 层文件

路径：

```text
sessions/SESSION_CODE/
```

Session 层只放两类共享事实：

- 这个 session 的实验配置和阶段状态。
- 这个 session 的随机化与公司分配日志。

不要在 session 层找某个被试的答题内容；答题内容都在 participant 目录下。

### 3.1 session_metadata.json

用途：理解 session 的基本状态和配置快照。

字段：

| 字段 | 含义 | 数据来源 |
|---|---|---|
| `sessionId` | session 内部 ID | `Session.id` |
| `sessionCode` | 被试进入时使用的 session code | `Session.code` |
| `createdAt` | session 创建时间 | `Session.createdAt` |
| `completedAt` | session 完成时间；未完成为 `null` | `Session.updatedAt` + `Session.status` |
| `status` | `WAITING` / `MATCHED` / `IN_PROGRESS` / `COMPLETED` | `Session.status` |
| `runtimePhase` | 当前运行阶段 | `Session.runtimePhase` |
| `durations.practiceMinutes` | 测试轮时长 | `SessionSegmentState` |
| `durations.workMinutes` | 正式工作段时长 | `SessionSegmentState` |
| `durations.breakMinutes` | 休息段时长 | `SessionSegmentState` |
| `durations.aCompanyWindowSeconds` | 尽调员单家公司窗口，当前固定 300 秒 | 导出服务 |
| `experimentConfigSnapshot.activeExperimentMode` | 创建 session 时的实验模式 | `Session.experimentMode` |
| `experimentConfigSnapshot.snapshot` | 创建 session 时固化的实验配置 | `Session.experimentSnapshot` |

常用判断：

- 看 `status` 判断这个 session 是否完整。
- 看 `runtimePhase` 判断未完成 session 卡在哪一步。
- 看 `experimentConfigSnapshot` 判断 admin 后续修改是否影响这个 session。旧 session 不会被后续 admin 修改覆盖。

### 3.2 randomization.json

用途：看所有随机化结果和 seed。

字段：

| 字段 | 含义 | 数据来源 |
|---|---|---|
| `roleAssignment.method` | 角色随机方法 | `RandomizationAudit.roleAssignmentMethod` |
| `roleAssignment.seed` | 角色随机 seed | `RandomizationAudit.roleAssignmentSeed` |
| `roleAssignment.assignedAt` | 角色分配时间 | `RandomizationAudit.roleAssignedAt` |
| `roleAssignment.participantAId` | 内部 A，被试前台是尽调员 | `Pairing.participantAId` |
| `roleAssignment.participantBId` | 内部 B，被试前台是投资经理 | `Pairing.participantBId` |
| `companySequence.method` | 公司顺序随机方法 | `RandomizationAudit.companySequenceMethod` |
| `companySequence.seed` | 公司顺序 seed | `RandomizationAudit.companySequenceSeed` |
| `companySequence.generatedAt` | 公司顺序生成时间 | `RandomizationAudit.companySequenceGeneratedAt` |
| `companySequence.sequence` | session 共享正式公司顺序 | `RandomizationAudit.companySequence` |
| `bAssignment.method` | B 动态分配方法 | `RandomizationAudit.bAssignmentMethod` |
| `bAssignment.log` | B 每次从 locked pool / PreA 分配公司的日志 | `RandomizationAudit.bAssignmentLog` |
| `experiment` | 实验 1/2/3 或手动模式的处理变量 | `RandomizationAudit.experimentRandomization` 或 `Session.experimentSnapshot` |

实验处理变量重点看：

| 字段 | 含义 |
|---|---|
| `experiment.experimentMode` | `manual` / `ai_upgrade` / `side_reminder` / `coop_narrative` |
| `experiment.upgradeCohort` | 实验 1 的 early / late AI 升级组 |
| `experiment.sideDispatchMode` | 副线提醒模式：`continuous` / `batch` |
| `experiment.narrativeGroup` | 叙事组：`neutral_info` / `coop_narrative` |
| `experiment.themeOrder` | 合作叙事主题顺序 |
| `experiment.segmentAiStates` | 三个正式工作段的 AI 档位 |
| `experiment.fixedVariables` | 当前实验模式下被固定的变量 |
| `experiment.seeds` | 实验处理随机 seed |

## 4. Participant 层总览

路径：

```text
sessions/SESSION_CODE/participants/PARTICIPANT_ID/
```

一个 participant 目录就是一个被试的完整数据。A/B 分开，不互相覆盖。

### 4.1 participant_metadata.json

用途：看这个被试是谁、是什么角色、有没有完成关键阶段。

字段：

| 字段 | 含义 | 来源 |
|---|---|---|
| `participantId` | 被试内部 ID | `Participant.id` |
| `phone` | 登录手机号或准入令牌 | `Participant.phone` |
| `internalRole` | 内部角色：`A` / `B` | `Pairing` |
| `displayRole` | 前台角色：尽调员 / 投资经理 | 导出服务 |
| `partnerParticipantId` | 同 session 另一名被试 ID | `Pairing` |
| `loginAt` | 登录时间；旧数据可能回退为创建时间 | `ExperimentEvent.participant_login` / `Participant.createdAt` |
| `instructionViewedAt` | 看指导语时间 | `ExperimentEvent.instruction_viewed` |
| `practiceReadyAt` | 测试轮 ready 时间 | `ExperimentEvent.practice_ready` |
| `formalReadyAt` | 正式轮 ready 时间 | `ExperimentEvent.formal_ready` |
| `completedAt` | 完成时间；session 未完成则为 `null` | `Session.updatedAt` |
| `qualityFlags` | 数据质量旗标 | 导出服务 |

常见 `qualityFlags`：

| flag | 含义 |
|---|---|
| `session_not_completed` | session 未完整结束 |
| `missing_practice_quiz` | 缺测试题记录 |

### 4.2 variables.json

用途：快速分析用的摘要变量。它不是原始数据全集，原始事实仍以各文件为准。

字段：

| 字段 | 含义 |
|---|---|
| `schemaVersion` | 变量摘要版本 |
| `participantId` | 被试 ID |
| `sessionCode` | session code |
| `role` | 内部角色 A/B |
| `displayRole` | 前台角色 |
| `treatments` | 实验处理变量 |
| `mainline` | 主线摘要 |
| `ai` | AI 使用摘要 |
| `sideTask` | 副线摘要 |
| `questionnaire` | 问卷摘要 |
| `qualityFlags` | 数据质量旗标 |

#### treatments

| 字段 | 含义 | 可用于 |
|---|---|---|
| `experimentMode` | 当前 session 使用的实验模式 | 分实验 1/2/3 或手动模式 |
| `upgradeCohort` | AI 升级早/晚组 | 实验 1 处理变量 |
| `sideDispatchMode` | `continuous` / `batch` | 实验 2 处理变量 |
| `narrativeGroup` | `neutral_info` / `coop_narrative` | 实验 3 处理变量 |
| `themeOrder` | 叙事主题顺序 | 实验 3 主题顺序控制 |
| `segmentAiStates` | 工作段 1/2/3 AI 档位 | AI 档位暴露 |

#### mainline

| 字段 | 含义 |
|---|---|
| `companyCountTouched` | 该被试接触过的主线公司数 |
| `companyCountCompleted` | 该被试完成的主线公司数 |
| `autoSubmittedCount` | 尽调员自动提交公司数；投资经理为 0 |
| `viewedAInfoCount` | 投资经理查看 A 信息次数；尽调员为 0 |
| `meanTimeToViewASeconds` | 投资经理从 A 信息解锁到查看的平均秒数；尽调员为 `null` |
| `meanBSubmitDelaySeconds` | 投资经理从 A 信息解锁到提交的平均秒数；尽调员为 `null` |

#### ai

| 字段 | 含义 |
|---|---|
| `mainAiRequestCount` | 主线 AI 用户请求次数 |
| `sideAiRequestCount` | 副线 AI 用户请求次数 |
| `basicRequestCount` | 基础版 AI 用户请求次数 |
| `advancedRequestCount` | 升级版 AI 用户请求次数 |
| `imageUploadCount` | 带图片附件的 AI 消息数 |

#### sideTask

| 字段 | 含义 |
|---|---|
| `plannedCount` | 该被试目录下副线计划总数 |
| `releasedCount` | 已释放 / 已展示到前端队列的题数 |
| `shownCount` | 当前等同于 `releasedCount` |
| `answeredCount` | 已作答题数 |
| `correctCount` | 作答正确题数 |
| `accuracy` | 正确率；无作答为 `null` |

#### questionnaire

| 字段 | 含义 |
|---|---|
| `practiceQuizPassed` | 是否有测试题作答记录 |
| `break1Submitted` | 休息问卷 1 是否提交 |
| `break2Submitted` | 休息问卷 2 是否提交 |

## 5. 问卷目录 questionnaires/

路径：

```text
participants/PARTICIPANT_ID/questionnaires/
```

### 5.1 practice_quiz.json

用途：测试题记录。

字段：

| 字段 | 含义 | 来源 |
|---|---|---|
| `templateId` | 测试题模板 ID | `QuestionnaireResponse.templateId` |
| `submittedAt` | 最后一次提交时间 | `QuestionnaireResponse.submittedAt` |
| `attempts` | 所有作答尝试 | `QuestionnaireResponse` |

`attempts[]` 字段：

| 字段 | 含义 |
|---|---|
| `attemptNo` | 第几次作答 |
| `submittedAt` | 本次提交时间 |
| `templateId` | 模板 ID |
| `answers` | 被试答案原始 JSON |
| `templateItems` | 当时模板题目 |

### 5.2 break_1.json / break_2.json

用途：两个正式工作段后的休息问卷。

字段：

| 字段 | 含义 |
|---|---|
| `templateId` | 问卷模板 ID |
| `segmentIndex` | 休息段 index：`2` 或 `4` |
| `workSegmentBeforeBreak` | 之前刚结束的工作段：`1` 或 `2` |
| `shownAt` | 休息段开始时间 |
| `submittedAt` | 提交时间；未交为 `null` |
| `answers` | 被试答案；未交为 `null` |
| `templateItems` | 当时模板题目 |
| `missing` | 是否缺失 |

## 6. 练习轮目录 practice_round/

路径：

```text
participants/PARTICIPANT_ID/practice_round/
```

练习轮不进入正式三段，但保留完整数据，便于排查学习成本和系统异常。

### 6.1 round_metadata.json

| 字段 | 含义 |
|---|---|
| `phase` | 固定为 `PRACTICE` |
| `segmentIndex` | 固定为 `0` |
| `startedAt` | 练习轮开始时间 |
| `endedAt` | 练习轮结束时间 |

### 6.2 companies/COMPANY_CODE__COMPANY_ID/

练习轮公司也按公司目录保存，文件和正式公司一样：

```text
company_metadata.json
answer_content.json
snapshots.jsonl
ai_chat.jsonl
```

### 6.3 side_tasks.jsonl

用途：练习轮副线题记录。结构和正式 `side_tasks/side_responses.jsonl` 相同，但只包含 `segmentIndex = 0` 的练习题。

### 6.4 side_ai_chat.jsonl

用途：练习轮副线 AI 聊天。结构和正式副线 AI 一样，但只包含 `segmentIndex = 0`。

## 7. 正式段目录 formal_segments/

路径：

```text
participants/PARTICIPANT_ID/formal_segments/
```

正式工作段映射：

| 文件夹 | workSegment | segmentIndex | 后续阶段 |
|---|---:|---:|---|
| `segment_1` | 1 | 1 | `break_1`，segmentIndex 2 |
| `segment_2` | 2 | 3 | `break_2`，segmentIndex 4 |
| `segment_3` | 3 | 5 | 结束页 |

### 7.1 segment_metadata.json

字段：

| 字段 | 含义 | 来源 |
|---|---|---|
| `workSegment` | 正式工作段编号：1/2/3 | 导出服务 |
| `segmentIndex` | runtime 阶段 index：1/3/5 | `SessionSegmentState.segmentIndex` |
| `startedAt` | 工作段开始时间 | `SessionSegmentState.startedAt` |
| `plannedEndsAt` | 计划结束时间 | `SessionSegmentState.endsAt` |
| `endedAt` | 实际结束时间 | `SessionSegmentState.completedAt` |
| `aiLevel` | 当前段 AI 档位 | session 快照 |
| `sideDispatchMode` | 当前段副线提醒模式 | `SideTaskSessionConfig` / session 快照 |
| `narrativeGroup` | 当前段叙事组 | `SideTaskSessionConfig` / session 快照 |
| `themeLabel` | 当前段合作叙事主题 | `SideTaskSessionConfig.segmentNTheme` |
| `companyIdsTouchedByParticipant` | 该被试本段接触过的公司 ID | `TaskAssignment` + `TaskSnapshot` |

## 8. 公司目录 companies/COMPANY_CODE__COMPANY_ID/

每个被试、每个公司、每个出现过的工作段都会有公司目录。

```text
company_metadata.json
answer_content.json
snapshots.jsonl
ai_chat.jsonl
```

### 8.1 company_metadata.json

用途：看这家公司是什么、怎么分配给被试、材料是什么、关键时间点是什么、AI 状态是什么。

#### company

| 字段 | 含义 | 来源 |
|---|---|---|
| `companyId` | 公司内部 ID | `Company.id` |
| `companyCode` | 公司代号，通常来自 `roundLabel` | `Company.roundLabel` |
| `companyName` | 公司名称 | `Company.name` |
| `sequenceIndex` | A 端共享公司顺序中的位置 | `TaskAssignment.sequenceIndex` |
| `materialVersion` | 材料版本 | `Company.materials.metadata.version` 或默认 `v1` |

#### task

| 字段 | 含义 |
|---|---|
| `taskAssignmentId` | 这家公司在该 session 的任务 ID |
| `phase` | `PRACTICE` / `FORMAL` |
| `workSegmentFirstSeen` | 该被试第一次接触该公司的工作段 |
| `internalRole` | A/B |
| `displayRole` | 尽调员 / 投资经理 |
| `bSequenceIndex` | 投资经理处理顺序；尽调员通常为 `null` |
| `bAssignmentPath` | B 公司分配路径：locked pool / PreA fallback 等 |
| `bAssignmentDetail` | B 分配时的候选池、seed、选中公司等完整日志 |

`bAssignmentDetail` 常用字段：

| 字段 | 含义 |
|---|---|
| `assignmentPath` / `method` | 分配路径 |
| `assignedAt` | B 获得公司时间 |
| `eligibleCompanyIds` | 当时可选公司池 |
| `lockedPoolSize` | 锁定池大小 |
| `seed` / `assignmentSeed` | 分配随机 seed |
| `chosenCompanyId` | 最终选中公司 |
| `preAActiveCompanyId` | PreA fallback 时 A 正在处理的公司 |

#### materials

| 字段 | 含义 |
|---|---|
| `companyId` | 公司 ID |
| `materialVersion` | 材料版本 |
| `visibleMaterialIds` | 该被试实际可见材料 ID |
| `visibleMaterialNames` | 该被试实际可见材料名 |
| `visibleScopes` | 可见材料范围：`shared` / `diligence` / `manager` |
| `diligenceMaterialIds` | 尽调员专属材料 ID |
| `managerMaterialIds` | 投资经理专属材料 ID |
| `researchOnlyMaterialIds` | 研究者材料 ID，不直接给被试看 |
| `storageKeys` | 可见材料存储索引 |

注意：线上数据库里不一定有“本地目录”概念，所以导出记录材料 ID、版本、scope、storageKey。研究者需要回查材料时，用 `companyId + materialVersion + storageKeys`。

#### timing

| 字段 | 含义 |
|---|---|
| `aStartedAt` | A 开始处理该公司的时间 |
| `aDeadlineAt` | A 5 分钟窗口截止时间 |
| `aSubmittedAt` | A 提交时间，通常是自动提交 |
| `aUnlockedForBAt` | A 信息解锁给 B 的时间 |
| `bAssignedAt` | B 获得该公司的时间 |
| `bViewedAInfoAt` | B 点击查看 A 信息的时间 |
| `bCompletedAt` | B 提交该公司的时间 |

#### aiState

| 字段 | 含义 |

### 8.2 answer_content.json

用途：保存该被试对该公司的答题区全文。

字段：

| 字段 | 含义 | 来源 |
|---|---|---|

### 9.2 side_responses.jsonl

用途：每道副线题一行，包含题目、呈现、作答、正确率、反应时。

字段：

| 字段 | 含义 |
|---|---|
| `participantId` | 被试 ID |
| `planId` | 副线计划 ID |
| `itemId` | 副线题目 ID |
| `itemCode` | 题目代号 |
| `workSegment` | 工作段 |
| `prompt` | 题目材料文本 |
| `question` | 问题 |
| `options` | 选项数组 |
| `goldAnswer` | 正确答案键，通常为 `A` / `B` |
| `correctAnswerText` | 正确答案全文 |
| `shownAt` | 题目释放 / 显示时间 |

| 想看的东西 | 优先文件 | 备注 |

## 14. JSONL 阅读方法

JSONL 不是一个大 JSON 数组，而是每一行一个 JSON。

适合直接按行读：

```text
第 1 行：一次事件 / 一条 AI 消息 / 一道副线题
第 2 行：下一次事件 / 下一条 AI 消息 / 下一道副线题
...
```

研究时常见处理：

- 按 `createdAt` / `eventAt` / `serverTime` 排序还原时间线。
- 按 `requestId` 合并 AI 的 user / assistant。
- 按 `planId` 合并副线计划、事件、作答。
- 按 `taskAssignmentId` 合并公司答题、AI、事件、快照。

## 15. 当前第一期边界

第一期已经覆盖的原始数据：

- session 元数据、配置快照、随机化 seed。
- A/B participant 元数据、角色、登录、ready、完成状态。
- 测试题、两段休息问卷。
- 练习轮公司、答题、AI、副线。
- 正式工作段 1/2/3 的时间、AI 档位、副线模式、叙事主题。
- 每家公司材料索引、A/B 答题全文、B feedback、快照。
- A 5 分钟窗口、自动提交、B 查看 A、B 提交。
- B locked pool / PreA 分配路径和候选池日志。
- 主线 AI、副线 AI、AI 档位、模型、耗时、错误、图片附件。
- 副线计划、释放、提醒、打开、作答、正确率、归档。
- `variables.json` 中的核心处理变量和常用摘要。

第一期不直接存成论文变量、但可后处理：

- AI 采纳率：需要 AI 聊天 + 最终文本 + 相似度或人工编码。
- 内容质量：需要评分规则和评分员。
- gold fact / gold issue：需要研究者维护公司金标准。
- 复杂注意力变量：滚动、focus/blur、复制粘贴、细粒度停留时间目前不作为第一期核心记录。

## 16. 快速审查清单

拿到一个新导出包后，可以按这个顺序检查：

1. `manifest.json` 的 session / participant 数量是否符合实验人数。
2. 每个 `sessions/SESSION_CODE/` 是否有 `session_metadata.json` 和 `randomization.json`。
3. 每个 session 下是否有两个 participant 目录。
4. 每个 participant 是否有 `participant_metadata.json`、`variables.json`。
5. `questionnaires/` 是否有 `practice_quiz.json`、`break_1.json`、`break_2.json`。
6. `practice_round/companies/` 是否有公司目录。
7. `formal_segments/segment_1/2/3/segment_metadata.json` 是否存在。
8. 接触过的公司目录是否有 `company_metadata.json`、`answer_content.json`、`snapshots.jsonl`、`ai_chat.jsonl`。
9. `side_tasks/side_plan.json` 是否有计划题。
10. `side_tasks/side_responses.jsonl` 是否能看到未答和已答题。
11. `events/events.jsonl` 是否有登录、ready、问卷、保存、AI 等关键事件。
12. 如果使用图片，`ai_chat.jsonl.attachments[].relativePath` 是否能在 `attachments/images/` 找到文件。
13. `variable_implementation_summary.json.missingSource` 是否为空；如不为空，优先处理。
14. `risks` 若只来自旧 session 或未选具体副线题的 side AI，通常是兼容风险，不一定影响新 session 核心变量。
