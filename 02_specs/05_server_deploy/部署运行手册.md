## 1. 文件说明

- `02_specs/05_server_deploy/命令运行清单.md`：最重要的执行入口，集中列出你后续要运行的所有命令。
- `apps/web/Dockerfile`：构建并启动 Next.js 前端。
- `apps/server/Dockerfile`：构建并启动 NestJS 后端。
- `apps/server/docker-entrypoint.sh`：后端容器启动前执行 Prisma migration。
- `compose.production.yml`：生产式单机 compose。
- `infra/nginx/production.conf`：生产 Nginx 反向代理配置，负责 80/443、`/api/*` 反代和域名跳转。
- `.env.production.example`：生产环境变量模板。
- `.dockerignore`：减少 Docker build 上下文，并排除私钥和本地数据。
- `scripts/deploy/check-server.ps1`：本地通过 SSH 检查服务器状态。
- `scripts/deploy/fix-ssh-key-permissions.ps1`：修复 Windows OpenSSH 私钥权限太开放的问题。
- `scripts/deploy/probe-ssh-users.ps1`：私钥权限正常但 publickey 登录失败时，自动试探常见服务器用户名。
- `scripts/deploy/upload-git-archive.ps1`：当前推荐部署脚本，本地从 git HEAD 生成归档并上传服务器。
- `scripts/deploy/upload-project.ps1`：旧本地整包上传脚本，不优先使用。
- `scripts/deploy/sync-from-github.sh`：服务器从 GitHub 拉取最新源码并同步到 `/opt/multi-cooperation`，当前降级为可选路线。
- `scripts/deploy/create-prod-env.ps1`：本地读取基础/高级 AI 配置，并上传服务器 `.env.production`；首次部署会生成数据库密码，后续重跑默认复用服务器已有密码。
- `scripts/deploy/prepare-server.sh`：服务器初始化 Docker 与基础目录。
- `scripts/deploy/deploy-prod.sh`：服务器上构建、启动并检查生产服务。
- `scripts/deploy/show-prod-logs.sh`：服务器上查看 compose 服务日志。

## 2. 推荐协作方式

部署和环境配置很慢，后续默认用带进度输出的脚本推进。

本地先执行：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/deploy/check-server.ps1
```

服务器初始化脚本、生产部署脚本都会输出类似：

```text
==> [1/8] Checking deployment files
==> [2/8] Showing compose plan
==> [3/8] Building images
```

如果卡住，优先把最后一个阶段标题和其后的错误日志贴回来。

## 3. 本地生产式验证

在根目录创建本地测试用 `.env.production`：

```powershell
Copy-Item .env.production.example .env.production
```

至少修改：

```text
POSTGRES_PASSWORD=一个长随机密码
NEXT_PUBLIC_SERVER_BASE_URL=http://localhost:3001
```

启动：

```powershell
docker compose --env-file .env.production -f compose.production.yml up -d --build
```

查看状态：

```powershell
docker compose --env-file .env.production -f compose.production.yml ps
docker compose --env-file .env.production -f compose.production.yml logs -f server
```

访问：

- web：`http://localhost:3000`
- server health：`http://localhost:3001/health`

停止：

```powershell
docker compose --env-file .env.production -f compose.production.yml down
```

注意：不要随意加 `-v`，否则会删除数据库和后端 storage volume。

## 5. 数据持久化

当前 compose 使用两个 Docker volume：

- `multi-cooperation_postgres_data`：PostgreSQL 数据库。保存 session、participant、randomization、任务分配、副线作答、AI 聊天索引、实验事件、导出任务状态等结构化实验数据。
- `multi-cooperation_server_storage`：后端运行时文件存储。容器内路径为 `/app/storage`，由 `STORAGE_ROOT=/app/storage` 指定，包括材料运行时副本、AI 图片附件、变量导出包、临时上传文件。

不要混淆两个材料位置：

```text
/opt/multi-cooperation/00_start_materials/原始材料
```

这是 admin 扫描用的源材料库。

```text
/app/storage/materials
```

这是导入后供前台实际访问的运行时材料目录，背后是 `multi-cooperation_server_storage` volume。

重要纪律：

- `docker compose down` 不会删除 volume。
- `docker compose down -v` 会删除 volume，正式环境不要执行。
- 不要执行 `docker volume rm multi-cooperation_postgres_data` 或 `docker volume rm multi-cooperation_server_storage`。
- 不要手动删除 `/var/lib/docker/volumes/...` 下的生产 volume。
- 迁移服务器前必须同时备份 PostgreSQL 数据库和 `multi-cooperation_server_storage`。

## 6. 常用运维命令

```bash

```

看后端日志：

```bash

```

```bash

```

```bash

```

重新构建并启动：

```bash
docker compose --env-file .env.production -f compose.production.yml up -d --build
```

## 7. HTTPS / Nginx 入口

当前公网主入口已经是：

- `https://aiseek.tech`
- `https://aiseek.tech/admin`
- `https://aiseek.tech/api/health`

Nginx 负责：

- `http://aiseek.tech` 301 到 `https://aiseek.tech/`。
- `https://www.aiseek.tech` 301 到 `https://aiseek.tech/`。
- `/` 反代到 web。
- `/api/*` 去掉 `/api` 前缀后反代到 server。

证书文件在服务器 `/etc/multi-cooperation/certs`，私钥不进 git。

完整浏览器 smoke 通过前，3000 / 3001 可暂时保留为回退端口；正式收口后安全组只保留 `80 / 443 / 22`。

## 9. 长期部署协作方式修订（2026-06-17）

后续默认把 Codex 的职责放在本地：读代码、改代码、跑本地验证、记录文档、commit、push。线上部署不要再依赖 Codex 每次手动 SSH 到服务器调试 GitHub clone。

当前推荐方式是：

```powershell
cd "E:\Own_program\multi cooperation"
powershell -ExecutionPolicy Bypass -File scripts/deploy/upload-git-archive.ps1 -Service web
```

根据影响面选择：

- `-Service web`：纯前端修改。
- `-Service server`：后端、数据库、导出、实验流程逻辑修改。
- `-Service all`：前后端都有修改，或不确定影响面。

这个方式比服务器直接 `git clone` 更适合当前环境，因为服务器访问 GitHub 443 端口不稳定；本地到服务器的 SSH/SCP 相对可控。GitHub 仍作为代码历史与备份真相源，但不再要求服务器每次都能直连 GitHub。

长期更完整的 P1/P2 方案是 GitHub Actions 或 self-hosted runner：

- 本地/AI 只负责 push。
- CI 自动构建、测试和部署。
- 服务器只接收确定的构建产物或执行固定部署脚本。

这需要额外配置 GitHub Secrets、服务器 deploy key 或 runner，当前未作为本轮必需项。
