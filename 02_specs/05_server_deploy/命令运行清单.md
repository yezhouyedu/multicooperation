## 2. 查看服务状态

```bash
cd /opt/multi-cooperation
docker compose --env-file .env.production -f compose.production.yml ps
```

这条命令只负责“查看当前线上服务状态”，不会修改代码、不会重启服务、不会改数据库。

它会读取当前目录下的：

- `.env.production`：生产环境变量，比如数据库密码、AI key、端口。
- `compose.production.yml`：生产 Docker 编排文件。

然后显示四个核心容器是否正在运行：

- `postgres`：线上数据库。
- `server`：后端 API、实验逻辑、变量记录、导出服务。
- `web`：前端网页。
- `nginx`：公网 80/443 入口，负责 HTTPS、301 跳转和 `/api/*` 反代。

正常应看到四个容器都是 `Up` 状态：

```
multi_cooperation_nginx_prod      nginx:1.27-alpine       Up (healthy)
multi_cooperation_postgres_prod   postgres:16          Up (healthy)
multi_cooperation_server_prod     multi-cooperation-server   Up (healthy)
multi_cooperation_web_prod        multi-cooperation-web      Up
```

如果提示没有 Docker 权限，命令前加 `sudo`。

## 6. 重新部署（代码更新后）

重新部署前建议先做一次备份，尤其是改 Prisma migration、导出、附件、材料存储相关代码时。

### 方式 A：本地 Git archive 上传（当前推荐）

当本地已经 commit 并 push 到 GitHub 后，优先从本地上传当前 git HEAD：

```powershell
cd "E:\Own_program\multi cooperation"

# 纯前端修复
powershell -ExecutionPolicy Bypass -File scripts/deploy/upload-git-archive.ps1 -Service web

# 后端、数据库、导出、实验流程逻辑修复
powershell -ExecutionPolicy Bypass -File scripts/deploy/upload-git-archive.ps1 -Service server

# 前后端都有修改，或不确定影响面
powershell -ExecutionPolicy Bypass -File scripts/deploy/upload-git-archive.ps1 -Service all
```

这个方式只上传 `git archive HEAD`，不会上传本地未提交文件、导出包、node_modules 或证书私钥。服务器直连 GitHub 不稳定时，优先使用这个路线。

### 方式 B：GitHub 拉取部署（可选）

如果服务器到 GitHub 网络稳定，仍可在服务器运行：

```bash
cd /opt/multi-cooperation
sudo bash scripts/deploy/sync-from-github.sh
sudo bash scripts/deploy/deploy-prod.sh all
```

注意：如果 GitHub 仓库不是公开可读，服务器需要先配置 GitHub 凭据或 deploy key。否则 `sync-from-github.sh` 可能报 `could not read Username for 'https://github.com'`；如果服务器直连 GitHub 443 超时，也不要反复重试，直接切回方式 A。

这三步分别做什么：

| 命令 | 做什么 | 会不会动数据库数据 |
|------|--------|-------------------|
| `cd /opt/multi-cooperation` | 进入服务器上的项目目录。后续命令都要在这里执行。 | 不会 |
| `sudo bash scripts/deploy/sync-from-github.sh` | 从 GitHub 拉取最新 `main` 代码到临时目录，再同步到 `/opt/multi-cooperation`。会保留服务器上的 `.env.production`、`storage` 等生产数据目录。 | 不会主动改数据库 |
| `sudo bash scripts/deploy/deploy-prod.sh all` | 按生产配置重新构建 web/server 镜像，启动或更新容器，并执行 Prisma migration。 | 不会清空数据库；但如果有新 migration，会按 migration 修改表结构 |

运行完后建议立刻检查状态：

```bash
docker compose --env-file .env.production -f compose.production.yml ps
curl http://127.0.0.1:3001/health
```

如果 `deploy-prod.sh` 失败，不要反复盲目重跑。先看：

```bash
docker compose --env-file .env.production -f compose.production.yml logs --tail=80 server
```

### 方式 C：旧本地整包上传代码（不优先）

```powershell
cd "E:\Own_program\multi cooperation"
powershell -ExecutionPolicy Bypass -File scripts/deploy/upload-project.ps1 -User ubuntu
```

**步骤 2：服务器重新构建并启动**

```bash

sudo bash scripts/deploy/deploy-prod.sh
```

**补充：更新生产环境变量**

仅当 AI key、模型、端口或 API 地址变化时运行。脚本会优先复用服务器已有 PostgreSQL 密码，不会默认轮换数据库密码。


## 7. 数据库操作

### 7.1 查看当前持久化数据位置和大小

查看 Docker volumes：

```bash
docker volume ls | grep multi-cooperation
```

查看数据库和后端文件存储大小：

```bash
sudo du -sh /var/lib/docker/volumes/multi-cooperation_postgres_data/_data
sudo du -sh /var/lib/docker/volumes/multi-cooperation_server_storage/_data
```

查看 server 容器里实际使用的 storage 路径：

```bash
docker exec multi_cooperation_server_prod sh -lc 'echo STORAGE_ROOT=$STORAGE_ROOT; du -sh /app/storage; test -w /app/storage/exports && echo exports-writable; test -w /app/storage/attachments && echo attachments-writable'
```

当前口径：

- PostgreSQL 数据在 `multi-cooperation_postgres_data`。
- 材料运行时副本、AI 图片附件、变量导出包在 `multi-cooperation_server_storage`，容器内路径为 `/app/storage`。
- 源材料库在 `/opt/multi-cooperation/00_start_materials/原始材料`，不是运行时 storage。
- 生产 compose 将 `./00_start_materials` 只读挂载到 server 容器 `/app/00_start_materials`，admin 材料导入实际扫描 `/app/00_start_materials/原始材料`。

不要执行：

```bash
docker compose down -v
docker volume rm multi-cooperation_postgres_data
docker volume rm multi-cooperation_server_storage
```

```bash

```

**查看磁盘空间：**

```bash
df -h
```

**查看内存：**

```bash
free -h
```

**查看容器详细信息：**

```bash
docker inspect multi_cooperation_server_prod | tail -30
```

**如果服务打不开，按顺序检查：**

1. 容器是否在运行：`docker compose ps`
2. 后端是否健康：`curl http://localhost:3001/health`
3. 前端是否响应：`curl -o /dev/null -s -w "%{http_code}" http://localhost:3000`
4. 看日志找错误：`docker compose logs --tail=30 server`
5. 检查公网入口：`curl -I https://aiseek.tech/api/health`、`curl -I https://aiseek.tech/login`
6. 检查防火墙：正式入口至少放行 80/443/22；3000/3001 仅作为完整 HTTPS smoke 前的回退端口。

## 10. 2026-06-17 长期部署链路修订

### 10.1 当前推荐路线：本地 Git archive 上传

由于裸 IP 服务器访问 GitHub 443 端口多次超时，当前长期推荐路线改为：

1. 本地完成修改、构建验证、commit、push GitHub。
2. 本地用当前 Git commit 生成源码归档并上传服务器。
3. 服务器只重建需要变化的服务。

常用命令：

```powershell
cd "E:\Own_program\multi cooperation"

# 纯前端修复，只重建 web
powershell -ExecutionPolicy Bypass -File scripts/deploy/upload-git-archive.ps1 -Service web

# 后端或数据库 schema/migration 相关修复，重建 server
powershell -ExecutionPolicy Bypass -File scripts/deploy/upload-git-archive.ps1 -Service server

# 前后端都改了，或不确定影响面时
powershell -ExecutionPolicy Bypass -File scripts/deploy/upload-git-archive.ps1 -Service all
```

这个脚本的原则：

- 默认 SSH 用户为 `ubuntu`。
- 只上传 `git archive HEAD`，也就是当前已提交源码，避免把本地临时文件、导出包、node_modules、私钥一起传上去。
- 服务器同步时保留 `.env.production`、`00_start_materials/`、`storage/`、`apps/server/storage/`。
- 自动把服务器上的 `.sh` 和 `apps/server/docker-entrypoint.sh` 转成 Linux LF 行尾，避免 CRLF 导致 `pipefail\r` 或 entrypoint 无法启动。
- 默认会继续执行 `deploy-prod.sh`；如只想上传源码、不部署，可以加 `-SkipDeploy`。

### 10.2 GitHub 服务器直连路线降级为可选

如果服务器到 GitHub 网络稳定，仍可使用：

```bash
cd /opt/multi-cooperation
sudo bash scripts/deploy/sync-from-github.sh
sudo bash scripts/deploy/deploy-prod.sh all
```

但如果出现以下错误，不要反复重试：

```text
Failed to connect to github.com port 443
could not read Username for 'https://github.com'
```

遇到这类问题直接切回 `upload-git-archive.ps1`。

### 10.3 定向部署

`deploy-prod.sh` 现在支持可选参数：

```bash
sudo bash scripts/deploy/deploy-prod.sh web
sudo bash scripts/deploy/deploy-prod.sh server
sudo bash scripts/deploy/deploy-prod.sh all
```

- `web`：只构建并重启前端容器，适合页面、交互、样式、前端文案修复。
- `server`：只构建并重启后端容器，适合 API、实验逻辑、导出、Prisma migration 相关修复。
- `all`：完整构建并重启，适合前后端都变更，或影响面不确定时使用。

无论哪一种，都不要执行 `docker compose down -v`。
