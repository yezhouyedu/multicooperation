## 2. 查看服务状态

```bash
cd /opt/multi-cooperation
docker compose --env-file .env.production -f compose.production.yml ps
```

这条命令只负责“查看当前线上服务状态”，不会修改代码、不会重启服务、不会改数据库。

它会读取当前目录下的：

- `.env.production`：生产环境变量，比如数据库密码、AI key、端口。
- `compose.production.yml`：生产 Docker 编排文件。

然后显示三个核心容器是否正在运行：

- `postgres`：线上数据库。
- `server`：后端 API、实验逻辑、变量记录、导出服务。
- `web`：前端网页。

正常应看到三个容器都是 `Up` 状态：

```
multi_cooperation_postgres_prod   postgres:16          Up (healthy)
multi_cooperation_server_prod     multi-cooperation-server   Up (healthy)
multi_cooperation_web_prod        multi-cooperation-web      Up
```

如果提示没有 Docker 权限，命令前加 `sudo`。

## 3. 查看日志

**后端日志：**

```bash
cd /opt/multi-cooperation
docker compose --env-file .env.production -f compose.production.yml logs --tail=50 server
```

**前端日志：**

```bash
cd /opt/multi-cooperation
docker compose --env-file .env.production -f compose.production.yml logs --tail=50 web
```

**数据库日志：**

```bash
cd /opt/multi-cooperation
docker compose --env-file .env.production -f compose.production.yml logs --tail=50 postgres
```

**实时跟踪日志（加 -f）：**

```bash
docker compose --env-file .env.production -f compose.production.yml logs -f server
```

按 `Ctrl + C` 退出跟踪。

**重启全部：**



## 6. 重新部署（代码更新后）

重新部署前建议先做一次备份，尤其是改 Prisma migration、导出、附件、材料存储相关代码时。

### 方式 A：GitHub 拉取部署（推荐）

当本地已经 commit 并 push 到 GitHub 后，优先在服务器运行：

```bash
cd /opt/multi-cooperation
sudo bash scripts/deploy/sync-from-github.sh
sudo bash scripts/deploy/deploy-prod.sh
```

这个方式不走本地到服务器的大包 SSH 上传，适合网络不稳定或项目包较大时使用。

注意：如果 GitHub 仓库不是公开可读，服务器需要先配置 GitHub 凭据或 deploy key。否则 `sync-from-github.sh` 可能报 `could not read Username for 'https://github.com'`。遇到这种情况，临时小代码补丁可以继续走 SSH 上传；大材料文件建议用云控制台文件管理上传。

这三步分别做什么：

| 命令 | 做什么 | 会不会动数据库数据 |
|------|--------|-------------------|
| `cd /opt/multi-cooperation` | 进入服务器上的项目目录。后续命令都要在这里执行。 | 不会 |
| `sudo bash scripts/deploy/sync-from-github.sh` | 从 GitHub 拉取最新 `main` 代码到临时目录，再同步到 `/opt/multi-cooperation`。会保留服务器上的 `.env.production`、`storage` 等生产数据目录。 | 不会主动改数据库 |
| `sudo bash scripts/deploy/deploy-prod.sh` | 按生产配置重新构建 web/server 镜像，启动或更新容器，并执行 Prisma migration。 | 不会清空数据库；但如果有新 migration，会按 migration 修改表结构 |

运行完后建议立刻检查状态：

```bash
docker compose --env-file .env.production -f compose.production.yml ps
curl http://127.0.0.1:3001/health
```

如果 `deploy-prod.sh` 失败，不要反复盲目重跑。先看：

```bash
docker compose --env-file .env.production -f compose.production.yml logs --tail=80 server
```

## 7. 数据库操作

**进入数据库命令行：**

```bash
cd /opt/multi-cooperation
docker compose --env-file .env.production -f compose.production.yml exec postgres psql -U postgres -d multi_cooperation
```

**常用 SQL：**

```sql
-- 查看所有表
\dt

-- 查看 session 数量
SELECT count(*) FROM "Session";

-- 查看被试数量
SELECT count(*) FROM "Participant";

-- 退出
\q
```

**备份数据库：**

```bash
cd /opt/multi-cooperation
docker compose --env-file .env.production -f compose.production.yml exec postgres pg_dump -U postgres multi_cooperation > /tmp/backup_$(date +%Y%m%d).sql
```

## 8. 排查问题

**查看容器资源占用：**

```bash
docker stats --no-stream
```

**查看磁盘空间：**

```bash
df -h
```

**查看内存：**

```bash
free -h
```

**查看容器详细信息：**

```bash
docker inspect multi_cooperation_server_prod | tail -30
```

**如果服务打不开，按顺序检查：**

1. 容器是否在运行：`docker compose ps`
2. 后端是否健康：`curl http://localhost:3001/health`
3. 前端是否响应：`curl -o /dev/null -s -w "%{http_code}" http://localhost:3000`
4. 看日志找错误：`docker compose logs --tail=30 server`
5. 检查防火墙：云控制台安全组是否放行 3000/3001
