## 1. 协作方式

当前采用“本地改代码，服务器跑部署”的协作方式。

- 代码、Dockerfile、compose、部署文档都在本地仓库修改并验证。
- 服务器只作为运行环境，不在服务器上临时改业务代码。
- SSH 私钥不放入 `02_specs/`、不进入 Git、不写入 Markdown。
- 当前推荐私钥路径继续使用：`C:\Users\ASUS\Desktop\first_try.pem`。
- 如果未来要移动私钥，只能放到仓库外，或放到仓库内 `.secrets/` 并确保 `.secrets/` 被 gitignore。

### 1.1 长耗时部署任务协作方式

Docker build、依赖安装、服务器初始化、数据库备份这类任务经常很慢。后续默认采用更强反馈的方式：

1. 先由协作者写好脚本或明确分步骤命令。
2. 每个阶段输出清晰进度，例如 `[1/6] 检查 Docker`、`[2/6] 构建 server 镜像`。
3. 用户可以自己运行脚本并盯终端进度。
4. 协作者根据用户贴回来的日志继续判断。
5. 除非需要实时排障，否则不优先让一个长命令在后台沉默运行很久。

这样做的目标不是少做验证，而是让部署过程可观察、可中断、可复跑。

## 2. 上线阶段目标

不要把“能打开网址”和“正式实验可放心采数”混在一起。

阶段拆分：

1. P0：服务器裸 IP 跑通。
2. P1：域名、HTTPS、反向代理和上线运维流程成型。
3. P2：正式实验前的数据安全、备份、彩排和排障体系收口。

## 3. P0：裸 IP 可部署

目标：在服务器上不用开发命令，能用生产方式启动 web、server、PostgreSQL，并完成一次实验主流程小样本验证。

必须完成：

- 前端 Dockerfile：构建 Next.js，并以生产模式启动。
- 后端 Dockerfile：构建 NestJS，生成 Prisma Client。
- 生产 compose：统一启动 `postgres / server / web`。
- 生产环境变量模板：明确数据库、server、web、AI 配置。
- 数据库 migration：server 启动前执行 `prisma migrate deploy`。
- 持久化 volume：
  - PostgreSQL 数据。
  - 后端 `storage/`，包含材料、图片附件、导出 zip。
- 健康检查：
  - server `GET /health`。
  - web 首页或 admin 页面能访问。
- 裸 IP 访问：
  - 临时暴露 web 端口。
  - server 端口可先暴露用于测试，P1 再收口到反向代理。

P0 验收：

- `docker compose up -d --build` 能启动完整服务。
- `server` 容器日志中 migration 成功或显示 no pending migrations。
- 浏览器能打开 web。
- 登录、配对、指导语、测试题、测试轮至少能走到正式 ready。
- admin 能打开并能看到 session。
- admin 能创建导出任务并下载 zip。
- 导出包中 `variable_implementation_summary.json.missingSource` 没有新 session 的关键缺失。

## 4. P1：域名与稳定访问

目标：用户不再通过 `:3000` / `:3001` 访问，而是通过备案后的域名和 HTTPS 访问。

必须完成：

- DNS 解析到服务器 IP。
- ICP 通过后配置反向代理。
- 选择一种反向代理：
  - Caddy：证书自动化更省心。
  - Nginx：更传统，但证书和配置步骤更多。
- 外部只开放：
  - `80`
  - `443`
  - `22` SSH
- 内部服务：
  - web 只给反向代理访问。
  - server 只给反向代理访问。
  - PostgreSQL 不对公网暴露。
- 配置生产 API 地址：
  - 前端请求应指向公网 HTTPS server 路径或同域代理路径。
- CORS 收口：
  - P0 可以宽松。
  - P1 应限制到正式域名。

P1 验收：

- `https://你的域名` 能打开前端。
- admin 能正常访问后端 API。
- 材料文件能加载。
- AI 能正常请求。
- 图片上传、粘贴、导出 zip 下载正常。
- 关闭公网 `3000 / 3001 / 5432` 后系统仍可使用。

## 5. P2：正式实验前运维

目标：实验正式开始前，即使服务器重启、误操作、导出包变大，也能恢复和排障。

必须完成：

- 数据库备份：
  - 每天至少一次 `pg_dump`。
  - 正式实验当天开始前手动备份一次。
  - 每批实验结束后手动备份一次。
- storage 备份：
  - 备份 `storage/materials`。
  - 备份 `storage/attachments`。
  - 备份 `storage/exports`。
- 恢复演练：
  - 至少做一次“从备份恢复到新库/临时库”的演练。
- 日志查看方法：
  - web 日志。
  - server 日志。
  - postgres 日志。
  - 反向代理日志。
- 重启恢复：
  - 服务器重启后 compose 能自动拉起服务。
  - PostgreSQL volume 不丢数据。
  - 后端 storage volume 不丢材料和导出包。
- 正式彩排：
  - 两名被试完整跑完三段。
  - admin 导出 zip。
  - 研究者按 `数据库文件夹手册.md` 审查导出包。
- 小规模压力检查：
  - 至少模拟多组 session 同时在线。
  - 观察 CPU、内存、磁盘和 server 日志。

P2 验收：

- 有可执行备份脚本。
- 有恢复说明。
- 有正式实验前 checklist。
- 新 session 导出包结构稳定。
- 项目维护者知道如何查看日志、重启服务、下载备份和确认数据存在。
