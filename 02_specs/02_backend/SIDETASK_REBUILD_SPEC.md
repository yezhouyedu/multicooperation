## 1. 先说结论

这份文档最初是从“副线原型升级计划”写起的；当前项目里该系统已经完成正式重构，参与者前台统一称为“任务2”，后端模型仍沿用 `SideTask*` 命名。

现在的正式实现包括：

- 顶部有一个“任务2”入口和低干扰滚动提醒
- 正式工作段按 session 级 `SideTaskPlan` 队列持续释放题目
- continuous / batch 只改变提醒频率，不改变后台题目实际到达节奏
- 任务2打开、释放、提醒、作答、AI 聊天、时间戳事件均可记录和导出
- 测试轮真实任务2数量固定为 0；教学演示题由前端临时注入，不进入 `SideTaskPlan`
- 有一个任务2 AI 面板，按“参与者 + session + 工作段”保留上下文

任务2不只是“有无副线”这么简单，而是承载至少两个实验操纵：

1. 实验局 2：副线提醒频率随机化
也就是同样总量的任务2题，到底是 `continuous` 每来一道就提醒，还是 `batch` 攒一段时间再提醒（后台题目实际到达速度一样）。

2. 实验局 3：合作叙事随机化  
也就是同样 40 条/段的任务2题里，团队到底看到的是：
- 合作叙事组：`20 条合作叙事 + 20 条普通中性`
- 对照组：`40 条普通中性`

所以接下来要建的不是“再美化一下副线条”，而是一整套正式副线系统。

## 2. 这次副线系统到底要控制什么

### 2.1 实验真正关心的变量

副线部分真正要控制的，不是题目具体长什么样，而是下面这几个变量：

#### 变量 A：副线提醒频率

只有两种：

- `continuous`
- `batch`

它控制的是：  
**同样 40 条题，后台都是每 30 秒到达一道，区别只是前端滚动提醒出现的频率。**
- `continuous`：每来一道就滚一次提醒（约每 30s 提醒一次）
- `batch`：攒约 5 分钟的题再提醒一次（参与者打开后会一次看到多道之前已到达但未被集中提醒的新题）

不是控制题目内容，也不是控制难度。

#### 变量 B：叙事组别

只有两种：

- `coop_narrative`
- `neutral_info`

它控制的是：  
**这一组团队整个实验期间，副线里是否掺入“合作叙事”内容。**

#### 变量 C：合作叙事主题顺序

只对 `coop_narrative` 组生效。

三类主题是：

- `互补分工`
- `验证留痕`
- `共同责任`

这三类主题要分布在三个工作段里，但顺序随机。  
也就是说，合作组不是固定“第一段互补、第二段留痕、第三段责任”，而是 6 种顺序之一。

#### 变量 D：单条题目是否允许直接问 AI

题库里已经有字段：

- `direct_ai_flag`

这不是一个新的实验局，但它是副线的重要题目属性。  
后面系统至少要能：

- 知道这条题是不是”可以直接丢给 AI 辅助”的题
- 在日志里保留这条题的这个属性

### 2.2 变量总览

```
实验设计（团队级别随机化，在 Session 创建时决定）
│
├── 变量A: 提醒频率 → continuous / batch （2水平）
│   └── 后台都是每30s到达一道，区别只是前端滚动提醒频率
│   └── continuous: 每道都提醒 / batch: 攒约5分钟提醒一次
│   └── 3个工作段保持一致，不逐段切换
│
├── 变量B: 叙事组别 → coop_narrative / neutral_info （2水平）
│   └── 决定该团队看到什么内容的副线题
│
├── 变量C: 主题顺序 → T1/T2/T3 的全排列 （6水平）
│   └── 仅对 coop_narrative 组生效
│   └── neutral_info 组此字段为空
│
└── 变量D: direct_ai_flag → 题目级属性
    └── 不是实验局，是每道题自带的标记
```

**随机化层级**：变量 A/B/C 都是**团队级**（session 级），不是个人级。同一个 session 里的两个参与者（A 角色和 B 角色）共享同一套实验条件。

**变量 A 的 UI 影响**：后台题目实际到达速度相同（每 30s 一道），区别在前端滚动提醒频率：
- `continuous`：每来一道就滚一次提醒（约每 30s 提醒一次），参与者感知"实时来件"
- `batch`：攒约 5 分钟的题再提醒一次（参与者打开后会看到多道之前已到达但未被集中提醒的新题），其余时间较安静

**变量 B/C 的 UI 影响**：页面结构完全不变，区别只在题目内容本身。参与者不会知道自己属于哪个组。

## 4. 还缺什么字段

Excel 这份题库已经够做“题库内容库”，但还不够做“实验运行系统”。

缺的不是题目文案字段，而是**分发层字段**和**暴露层字段**。

### 4.1 分发层缺口

题库只告诉我们“有哪些题”，但没有告诉我们：

- 这一组团队在第 1 段到底拿到哪 40 条
- 这 40 条里每条何时到达
- 到达顺序是什么
- 是不是本段里第 1 批 / 第 2 批 / 第 3 批

所以系统里必须新增“分发计划”这一层。

### 4.2 暴露层缺口

题库也没有记录：

- 这条题什么时候推给参与者
- 参与者有没有点开
- 什么时候回答
- 有没有用副线 AI
- 是否属于疑似 batch 粘贴处理

所以系统里也必须新增“副线事件日志”这一层。

## 5. 后端应该怎么拆

我建议不要再继续只用现在这张：

- `SideTaskResponseLog`

因为它太薄了，只能记“回答了什么”，记不住“这题是怎么被分发出来的”。

更合适的是拆成 4 层。

### 5.1 表一：`SideTaskItem`

这是题库主表。  
一行就是题库里的一道题。

建议字段：

- `id`
- `itemCode`
- `poolType`
- `workSegment`
- `surfaceScenario`
- `skeletonType`
- `narrativeCategory`
- `narrativeSubtype`
- `directAiFlag`
- `text`
- `question`
- `questionVariantId`
- `optionA`
- `optionB`
- `goldAnswer`
- `evidenceSpan`
- `distractorType`
- `distractorNote`
- `spilloverRiskFlag`
- `spilloverRiskNote`
- `difficulty`
- `version`
- `isActive`

这张表本质上就是把 Excel 正式入库。

### 5.2 表二：`SideTaskPlan`

这是“某个 session 某个工作段到底分到了哪些副线题”的计划表。  
一行不是题库定义，而是一次实验分发记录。

建议字段：

- `id`
- `sessionId`
- `segmentIndex`
- `itemId`
- `dispatchMode`
- `narrativeGroup`
- `themeLabel`
- `queueOrder`
- `batchNo`
- `scheduledAt`
- `releasedAt`
- `isArchivedAtSegmentEnd`

这张表解决的问题是：

**题库里有 900 道题，但这一组人这段到底拿到了哪 40 道？**

### 5.3 表三：`SideTaskExposureLog`

这是副线行为日志。  
每次推送、打开、回答、归档都可以记在这里。

建议字段：

- `id`
- `sessionId`
- `participantId`
- `sideTaskPlanId`
- `eventType`
- `eventAt`
- `payload`

优先要记的事件：

- `side_task_released`
- `side_task_opened`
- `side_task_answered`
- `side_task_archived`
- `side_task_ai_opened`
- `side_task_ai_sent`
- `side_task_bulk_candidate`

### 5.4 表四：`SideTaskSessionConfig`

这是“这个 session 的副线实验条件快照”。

建议字段：

- `sessionId`
- `dispatchMode`
- `narrativeGroup`
- `themeOrder`
- `segment1Theme`
- `segment2Theme`
- `segment3Theme`
- `segment1PlannedCount`
- `segment2PlannedCount`
- `segment3PlannedCount`
- `newsSequenceSeed`
- `distributionVersion`

这张表是为了后面写论文和追溯实验条件时不需要反推。

## 6. admin 到底需要改什么

admin 后面至少要能操作 4 类东西。

### 6.1 题库管理

要能：

- 导入正式题库 Excel
- 查看题目总量
- 按池子筛选
- 按工作段筛选
- 按叙事类别筛选
- 按 `direct_ai_flag` 筛选
- 按 `version` 筛选
- 停用单题

### 6.2 副线实验参数（admin 可调）

要能设置默认规则，例如：

- 每段总量是否固定 40
- 合作组每段是否固定 `20 合作 + 20 中性`
- 对照组每段是否固定 `40 中性`

#### 提醒频率参数（变量 A，admin 面板可调）

后台题目到达速度固定（每 30s 一道），区别只在前端滚动提醒频率。

**滚动提醒动画参数（两种模式共享）**：
| 参数 | 说明 | 默认值 |
|------|------|--------|
| `scroll_duration_sec` | 滚动条从右到左的动画时长 | 12 |
| `hold_sec` | 滚动条停留时长 | 5 |
| `fade_sec` | 滚动条淡出时长 | 2 |

**continuous 模式参数**：
| 参数 | 说明 | 默认值 |
|------|------|--------|
| `continuous_interval_sec` | 每道题之间的后台到达间隔秒数 | 30 |
| `continuous_jitter_sec` | 间隔的随机波动范围（±秒） | 0 |
| `continuous_pause_sec` | 淡出后到下一次提醒的暂停时长 | 15 |

**batch 模式参数**：
| 参数 | 说明 | 默认值 |
|------|------|--------|
| `batch_trigger_sec` | 每次提醒之间的间隔秒数（攒多久提醒一次） | 300 |
| `batch_pause_sec` | 淡出后到下一次提醒的暂停时长 | 60 |

> **设计意图**：后台到达速度两种模式一样（每 30s 一道），区别只在前端提醒频率。continuous 每来一道就提醒，batch 攒约 5 分钟提醒一次。admin 可以在同一个面板里分别调整两套参数。

### 6.3 实时 session 查看

要能看到某个 session：

- 当前副线实验条件
- 当前段已计划多少条
- 已释放多少条
- 已作答多少条
- 哪些题还在队列里

### 6.4 导出

要能导出：

- 题库定义
- 某个 session 的副线分发计划
- 某个 session 的副线行为日志

## 7. UI 应该怎么重构

### 7.1 顶部条不要大改

顶部任务2条保留低干扰入口思路：

- 左边固定 `任务2`
- 右边低干扰滚动提示 `任务2有新题目，请及时处理`

因为它负责的是“提醒你任务2来了”，不是正式做题。

### 7.2 真正要重做的是展开页

展开页不能再是“固定两题 + 上一题下一题”的 demo。

正式版应该是一个真正的副线收件箱。

建议结构：

#### 左侧：副线队列

显示：

- 已到达但未作答
- 已作答
- 当前选中题

不要只显示一题，要显示“队列感”。

#### 右上：题目详情与作答区

显示：

- 题干原文
- 问题
- 两个选项
- 当前题属性的小标签
  - 是否合作叙事
  - 是否允许直接 AI

#### 右下：副线 AI

保留独立上下文。

### 7.3 `continuous` 和 `batch` 在 UI 上怎么体现

重点不是做两套不同页面，而是让”提醒频率不同”。后台题目到达速度一样（每 30s 一道），区别只在前端滚动提醒出现的频率。

#### `continuous`

- 每来一道就滚一次提醒（约每 30s 提醒一次）
- 参与者感知”实时来件”

#### `batch`

- 攒约 5 分钟的题再提醒一次
- 参与者打开后看到多道新题
- 其余时间较安静

也就是说：

**页面长得差不多，题目到的速度也一样，但提醒频率不同。**

### 7.4 合作叙事组和中性组在 UI 上怎么体现

不要在页面上显式告诉参与者“你现在属于合作叙事组”。  
实验组别只能隐式体现在题目内容上。

所以：

- UI 结构不能变
- 标签不能暴露实验组
- 变化只在题目内容池和顺序上

## 8. 每段 40 条到底怎么做

这是最关键的产品规则。

### 8.1 当前建议规则

每个正式工作段都生成一份固定计划，共 `40` 条。

#### 中性对照组

每段：

- `40 条普通中性`

#### 合作叙事组

每段：

- `20 条合作叙事`
- `20 条普通中性`

其中这 `20 条合作叙事` 必须服从该段当前主题。  
例如这段轮到 `互补分工`，就只能从该主题对应池里抽。

### 8.2 与题库的对应关系

普通中性从：

- `pool_type = 普通中性池`
- `work_segment = 当前段`

合作叙事从：

- `pool_type = 合作叙事池`
- `work_segment = 当前段`
- `narrative_category = 当前段主题`

### 8.3 题目顺序怎么来

先抽集合，再在集合内部洗牌。  
洗牌必须有 seed，并保存。

这样后面可以回答：

- 这组人第 2 段到底看到哪 40 道
- 这些题的出现顺序是什么

### 8.4 分发计划示意图

#### 整体 Session 结构

│                                                             │
│  该团队的实验条件（Session 创建时随机决定）：                  │
│  ├── 提醒频率: continuous 或 batch（3个段保持一致）           │

#### continuous vs batch 提醒频率对比

后台到达速度相同（每 30s 一道），区别只在前端提醒频率：

```
Continuous（每条提醒）:
时间 ------------------------------------------------>
  提醒  提醒  提醒  提醒  提醒  提醒  ...  提醒
   *     *     *     *     *     *        *
  题1   题2   题3   题4   题5   题6  ...  题40
  每来一道就滚一次，参与者感知"实时来件"

Batch（分批提醒，约每5分钟提醒一次）:
时间 ------------------------------------------------>
  提醒                              提醒
   * *[==========================]   * *[====...
  一次提醒，但队列里已有多道题
  参与者打开后看到"一下来了好几道"
  实际上这些题是过去5分钟内每30s到一道攒出来的
```

## 11. 统计口径定义

### 11.1 totalReleased（已到达数）

**定义**：`releasedAt !== null` 的 plan 数量。

**为什么不用 `scheduledAt <= now`**：
- `scheduledAt` 是服务端预计算的"计划到达时间"，是理想值
- `releasedAt` 是前端真实首次在 sideTaskQueue 中检测到该 plan 后，POST `side_task_released` 到服务端，服务端用**服务端时间**写入的
- 两者可能有偏差（网络延迟、SSE 断线重连、前端检测延迟）
- 实验分析关心的是"参与者实际什么时候看到题"，不是"计划什么时候到"

**前端 SSE 断线恢复**：前端维护本地 `Set<string>` 记录已知 planId 集合。SSE 重连后对比新队列与本地集合，差集即"断线期间到达的题"，一次性补发 `side_task_released`。服务端幂等保证 `releasedAt` 不被覆盖。

### 11.2 totalAnswered（已回答数）

**定义**：当前 participant 有 `side_task_answered` ExposureLog 的 plan 数量。

**per-participant 而非 per-session**：
- 前端显示"已回答 X / 40"是当前参与者自己的进度
- 一个 session 只有一个 participant，所以 per-session 和 per-participant 结果一致
- 但代码按 per-participant 实现，为未来多人同 session 场景预留

**Admin 端**：Admin session 视图额外展示完整 plan 列表（不受 `scheduledAt <= now` 过滤），含每条的 scheduledAt、releasedAt、answered 状态（按 participant 过滤），用于实验监控。

### 11.3 totalPlanned（计划总数）

**定义**：当前段 SideTaskPlan 总数（固定 40）。

### 11.4 totalArchived（已归档未答题数）

**定义**：`isArchivedAtSegmentEnd = true` 且当前 participant 无 `side_task_answered` log 的 plan 数量。

### 11.5 队列显示 vs 统计的分离

| 用途 | 过滤条件 | 说明 |
|------|----------|------|
| 前端队列显示 | `scheduledAt <= now && !isArchivedAtSegmentEnd` | 显示"此刻应该看到的题" |
| totalReleased 统计 | `releasedAt !== null` | 统计"实际什么时候看到" |
| totalAnswered 统计 | participantId 过滤的 `side_task_answered` log | 统计"该参与者答了多少" |
| totalArchived 统计 | `isArchivedAtSegmentEnd && 无 answered log` | 统计"段结束时漏答多少" |
