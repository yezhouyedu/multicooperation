# Admin 材料库上传手册

这份手册对应当前项目的真实材料库口径。

目标只有一句话：
同一个公司案例下面，可以同时放

- 双方共享材料
- 只给尽调员看的材料
- 只给投资经理看的材料
- 只给研究者用的材料

admin 页面现在只负责两件事：

- 自动导入题库目录
- 选择公司并预览导入结果

不再推荐在后台逐个上传、替换、删除文件。

---

## 1. 现在的推荐目录结构

每个公司一个文件夹，例如 `P01/`、`P02/`。

推荐结构：

```text
00_start_materials/
  原始材料/
    P01/
      case.json
      participant/
        shared/
          1.公司简介.txt
          2.财务摘要.docx
        diligence/
          3.尽调员补充材料.docx
        manager/
          4.投资经理补充材料.docx
      research/
        0.信息点记录（研究者用）.txt
```

四层的意思：

- `participant/shared/`：尽调员和投资经理都能看到
- `participant/diligence/`：只发给尽调员
- `participant/manager/`：只发给投资经理
- `research/`：只给研究者和系统内部使用，不会发给参与者

---

## 2. 为什么要这么分

现在的要求是：

- 公司还是同一个公司 ID
- 但这个 ID 下面，尽调员和投资经理拿到的材料集合可以不同

所以系统层面必须支持：

- 同公司、同任务
- 不同角色看到不同材料

这次改完后，后端已经按这个逻辑导入和发放材料。

---

## 3. 系统怎么识别

当前扫描规则是：

1. `原始材料/` 下每个子文件夹都当成一个公司案例
2. 优先读取 `participant/` 和 `research/`
3. 在 `participant/` 里继续找三类子目录
4. 如果三类子目录不存在，就把 `participant/` 里的文件默认当成共享材料

也就是说：

```text
participant/
  1.公司简介.txt
  2.财务摘要.docx
```

这种老结构也还能导入，但会被当成：

- 全部都是共享材料

---

## 4. `case.json` 怎么写

`case.json` 不是强制，但建议保留。

示例：

```json
{
  "caseCode": "P01",
  "companyName": "高端装备样本 A",
  "roundLabel": "P01",
  "sector": "高端装备",
  "summary": "用于合作实验的高端装备案例。",
  "tags": ["P01", "高端装备"],
  "participantDir": "participant",
  "researchDir": "research",
  "sharedDir": "shared",
  "diligenceDir": "diligence",
  "managerDir": "manager",
  "autoFillSource": "research/0.信息点记录（研究者用）.txt",
  "sortOrder": 1
}
```

字段说明：

- `participantDir`：参与者材料总目录，默认是 `participant`
- `researchDir`：研究者目录，默认是 `research`
- `sharedDir`：共享材料目录，默认是 `shared`
- `diligenceDir`：尽调员材料目录，默认是 `diligence`
- `managerDir`：投资经理材料目录，默认是 `manager`
- `autoFillSource`：系统自动填充公司基础信息的来源文件，当前仍建议指向 `research/` 里的 txt

---

## 5. 导入后会发生什么

自动导入题库目录时，系统会：

1. 扫描每个公司文件夹
2. 复制文件到本地材料存储
3. 给每个文件打上元数据

当前关键元数据包括：

- `audience`
  - `participant` 或 `research`
- `participantRole`
  - `shared` / `A` / `B`

这里的 `A/B` 只是系统内部角色码：

- `A = 尽调员`
- `B = 投资经理`

参与者界面不显示这两个字母。

---

## 6. 运行时怎么发给两边

当前运行逻辑已经改成：

- 研究者材料：不发给参与者
- 共享材料：尽调员和投资经理都能看到
- 尽调员材料：只发给尽调员
- 投资经理材料：只发给投资经理

所以现在真的能实现：

- 同一个公司 ID
- 尽调员和投资经理看到不同材料

---

## 7. admin 页面现在保留什么

材料管理 tab 现在建议只保留这两个用途：

1. 点击“自动导入题库目录”
2. 选中某家公司，在右侧预览材料显示效果

不再把下面这些当主流程：

- 导入 P01 基线
- 在线编辑公司信息
- 在线上传单个材料
- 在线替换单个材料
- 在线删除单个材料

原因很简单：

- 实验材料本来就是成套维护
- 在线逐个改文件很容易乱
- 目录才是真正的真相源

---

## 8. 兼容旧结构

旧结构例如：

```text
P01/
  0.信息点记录（研究者用）.txt
  1.公司简介.txt
  2.财务摘要.docx
```

目前仍兼容。

但会按下面的方式理解：

- `信息点记录` 这类文件归到研究者材料
- 其他文件默认当成共享材料

也就是说，旧结构不能表达“只给尽调员”或“只给投资经理”。

要做角色差异材料，就必须改成新结构。

---

## 9. 对将来线上化的建议

你关心的点其实是：
现在本地读文件夹很顺手，但以后线上不可能让浏览器直接扫服务器磁盘，那怎么办？

建议分两阶段：

### 阶段 1：当前本地开发阶段

继续保留现在的“扫描本地题库目录”方案。

优点：

- 实现简单
- 便于快速迭代材料
- 不用先把对象存储、鉴权、上传流程一次性做完

### 阶段 2：线上正式版

改成“两层存储”：

1. 数据库存元数据
2. COS / S3 存文件本体

推荐分工：

- 数据库存：
  - 公司信息
  - 文件名
  - 文件类型
  - 排序
  - audience
  - participantRole
  - 文件 URL
  - 版本号
  - 上传批次
- COS 存：
  - pdf / docx / xlsx / txt 实体文件

这样做的原因：

- 数据库不适合存大量二进制文件
- 对象存储更适合大文件、预览、下载、版本迁移
- 后续要做网址导入、批量上传、回滚版本，都会更顺

---

## 10. “选择文件夹路径再读取”怎么做

如果以后是网页后台，严格说浏览器拿不到服务器任意路径。

实际可行方案通常是二选一：

### 方案 A：上传整个文件夹

前端支持选择目录，把目录里的文件批量上传到后端。

优点：

- 最符合浏览器安全模型
- 最容易接 COS

缺点：

- 一次上传很多文件时，要做进度、失败重传、批次记录

### 方案 B：后端配置一个固定导入目录

管理员把材料先放到服务器某个约定目录，再点“扫描导入”。

优点：

- 和你们现在本地开发习惯最接近

缺点：

- 只适合内网或运维可控环境
- 不适合通用 SaaS

如果以后要真正规范线上化，我建议最终走方案 A。

---

## 11. 当前最实用的工作流

现在最稳的流程就是：

1. 在线下整理好案例文件夹
2. 按新目录规范分成 `shared / diligence / manager / research`
3. 放进 `00_start_materials/原始材料/`
4. 回到 admin，点击“自动导入题库目录”
5. 在右侧预览检查导入结果

这就是当前版本的主流程。
