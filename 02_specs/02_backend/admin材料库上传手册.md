# Admin 材料库上传手册

这份手册对应当前项目的真实材料库口径。

目标只有一句话：
同一个公司案例下面，可以同时放

- 双方共享材料
- 只给 A 看的材料
- 只给 B 看的材料
- 只给研究者用的材料

admin 页面现在只负责两件事：

- 上传公司材料文件夹并替换入库材料
- 选择公司并在侧边栏预览导入结果

不再推荐在后台逐个上传、替换、删除文件。

---

## 1. 现在的推荐目录结构

现在推荐先分成两层案例库：

- `正式/`
- `测试轮/`

最终版材料库当前为：

- `正式/`：`P01` 到 `P36`，共 36 家正式公司。
- `测试轮/`：`P37`，共 1 家测试轮公司。

每个案例一个文件夹，例如 `P01/`、`P02/`、`P37/`。

推荐结构：

```text
00_start_materials/
  原始材料/
    正式/
      P01/
        case.json
        participant/
          shared/
            1.公司简介.txt
            2.财务摘要.docx
          diligence/
            3.A补充材料.docx
          manager/
            4.B补充材料.docx
        research/
          0.信息点记录（研究者用）.txt
    测试轮/
      P37/
        case.json
        participant/
          shared/
        research/
```

四层的意思：

- `participant/shared/`：A 和 B 都能看到
- `participant/diligence/`：只发给 A
- `participant/manager/`：只发给 B
- `research/`：只给研究者和系统内部使用，不会发给参与者

说明：`diligence` / `manager` 是历史目录名和内部材料分层标记；参与者前台统一显示 A / B，不再显示“尽调员 / 投资经理”。

---

## 2. 为什么要这么分

现在的要求是：

- 公司还是同一个公司 ID
- 但这个 ID 下面，A 和 B 拿到的材料集合可以不同

所以系统层面必须支持：

- 同公司、同任务
- 不同角色看到不同材料

这次改完后，后端已经按这个逻辑导入和发放材料。

---

## 3. 系统怎么识别

当前扫描规则是：

1. 优先扫描 `原始材料/正式/` 和 `原始材料/测试轮/`
2. 两个分层下的每个子文件夹都当成一个公司案例
3. 优先读取 `participant/` 和 `research/`
4. 在 `participant/` 里继续找三类子目录
5. 如果三类子目录不存在，就把 `participant/` 里的文件默认当成共享材料

兼容规则与当前红线：

- 如果 `原始材料/正式/` 不存在，旧的根级案例目录仍按正式案例处理。
- 如果 `原始材料/测试轮/` 不存在，旧版本曾临时回退使用 `P01` 作为测试轮案例；当前最终版必须提供 `测试轮/P37`，不要依赖回退口径。
- 2026-06-20 之后，导入器会把当前源目录中不存在的 `company-library-*` 公司标为 `legacy`；`legacy` 不进入 admin 材料预览列表，也不进入测试轮或正式轮派单池。
- 生产库已清理旧 `company-library-practice-p01` 与 `company-p01-baseline`。当前公司池应为正式 `P01-P36`、测试轮 `P37`，测试轮不得再取 P01。

也就是说：

```text
participant/
  1.公司简介.txt
  2.财务摘要.docx
```

这种老结构也还能导入，但会被当成：

- 全部都是共享材料

---

## 4. `case.json` 怎么写

`case.json` 不是强制，但建议保留。

示例：

```json
{
  "caseCode": "P01",
  "companyName": "高端装备样本 A",
  "roundLabel": "P01",
  "sector": "高端装备",
  "summary": "用于合作实验的高端装备案例。",
  "tags": ["P01", "高端装备"],
  "participantDir": "participant",
  "researchDir": "research",
  "sharedDir": "shared",
  "diligenceDir": "diligence",
  "managerDir": "manager",
  "usage": "formal",
  "autoFillSource": "research/0.信息点记录（研究者用）.txt",
  "sortOrder": 1
}
```

字段说明：

- `participantDir`：参与者材料总目录，默认是 `participant`
- `researchDir`：研究者目录，默认是 `research`
- `sharedDir`：共享材料目录，默认是 `shared`
- `diligenceDir`：A 材料目录，默认是 `diligence`
- `managerDir`：B 材料目录，默认是 `manager`
- `autoFillSource`：系统自动填充公司基础信息的来源文件，当前仍建议指向 `research/` 里的 txt
- `usage`：案例用途，`formal` 表示正式案例，`practice` 表示测试轮案例

## 5. 导入后会发生什么

通过 admin 浏览器文件夹上传或源目录导入时，系统会：

1. 扫描每个公司文件夹
2. 复制文件到本地材料存储
3. 给每个文件打上元数据

当前关键元数据包括：

- `audience`
  - `participant` 或 `research`
- `participantRole`
  - `shared` / `A` / `B`

这里的 `A/B` 与参与者前台显示一致；`diligence/manager` 只是材料目录名。

---

## 6. 运行时怎么发给两边

当前运行逻辑已经改成：

- 研究者材料：不发给参与者
- 共享材料：A 和 B 都能看到
- A 材料：只发给 A
- B 材料：只发给 B

所以现在真的能实现：

- 同一个公司 ID
- A 和 B 看到不同材料

---

## 7. admin 页面现在保留什么

材料管理 tab 现在只保留这几个正式用途：

1. 浏览当前已入库公司列表。
2. 勾选一个或多个公司，然后上传同数量的公司文件夹进行一一替换。
3. 上传包含 `正式/` 和 `测试轮/` 的大文件夹，执行全部替换。
4. 打开右侧预览栏，检查选中公司的材料显示效果。

不再把下面这些当主流程：

- 导入 P01 基线
- 在线编辑公司信息
- 在线上传单个材料
- 在线替换单个材料
- 在线删除单个材料
- 手工填写服务器本地路径后扫描

原因很简单：

- 实验材料本来就是成套维护
- 在线逐个改文件很容易乱
- 目录才是真正的真相源

当前上传校验规则：

- 每个公司文件夹必须能扫描到参与者材料和研究者材料。
- 推荐同时包含 `participant/shared`、`participant/diligence`、`participant/manager`、`research`。
- 若缺少 A 材料、B 材料、共享材料或研究者材料，后端会拒绝导入并在 admin 页面弹窗提示。
- 替换前，服务器会把旧的已入库材料文件备份到 `storage/material_backups/`；数据库中的公司记录和材料 metadata 随新上传材料更新。

## 8. 兼容旧结构

旧结构例如：

```text
P01/
  0.信息点记录（研究者用）.txt
  1.公司简介.txt
  2.财务摘要.docx
```

目前仍兼容。

但会按下面的方式理解：

- `信息点记录` 这类文件归到研究者材料
- 其他文件默认当成共享材料

也就是说，旧结构不能表达“只给尽调员”或“只给投资经理”。

要做角色差异材料，就必须改成新结构。

## 9. 对将来线上化的建议

你关心的点其实是：
现在本地读文件夹很顺手，但以后线上不可能让浏览器直接扫服务器磁盘，那怎么办？

建议分两阶段：

### 阶段 1：当前本地开发阶段

继续保留现在的“扫描本地题库目录”方案。

优点：

- 实现简单
- 便于快速迭代材料
- 不用先把对象存储、鉴权、上传流程一次性做完

### 阶段 2：线上正式版

改成“两层存储”：

1. 数据库存元数据
2. COS / S3 存文件本体

推荐分工：

- 数据库存：
  - 公司信息
  - 文件名
  - 文件类型
  - 排序
  - audience
  - participantRole
  - 文件 URL
  - 版本号
  - 上传批次
- COS 存：
  - pdf / docx / xlsx / txt 实体文件

这样做的原因：

- 数据库不适合存大量二进制文件
- 对象存储更适合大文件、预览、下载、版本迁移
- 后续要做网址导入、批量上传、回滚版本，都会更顺

## 10. “选择文件夹路径再读取”怎么做

如果以后是网页后台，严格说浏览器拿不到服务器任意路径。

实际可行方案通常是二选一：

### 方案 A：上传整个文件夹

前端支持选择目录，把目录里的文件批量上传到后端。

优点：

- 最符合浏览器安全模型
- 最容易接 COS

缺点：

- 一次上传很多文件时，要做进度、失败重传、批次记录

### 方案 B：后端配置一个固定导入目录

管理员把材料先放到服务器某个约定目录，再点“扫描导入”。

优点：

- 和你们现在本地开发习惯最接近

缺点：

- 只适合内网或运维可控环境
- 不适合通用 SaaS

如果以后要真正规范线上化，我建议最终走方案 A。

---

## 11. 当前最实用的工作流

现在最稳的流程就是：

1. 在线下整理好案例文件夹。
2. 按新目录规范分成 `shared / diligence / manager / research`
3. 如果只改少数公司，在 admin 材料管理 tab 勾选这些公司，点击“替换选中公司”，选择同数量的公司文件夹上传。
4. 如果要整体换库，点击“全部替换”，选择包含 `正式/` 和 `测试轮/` 的大文件夹上传。
5. 上传后在右侧预览检查导入结果。

这就是当前版本的主流程。
