## 1. 当前已确定的技术路线
本项目当前采用：

- 前端：**Next.js**
- 后端：**NestJS**
- 数据库：**PostgreSQL**
- 架构方式：**前后端分离**
- 项目组织：**monorepo**

这条路线的核心原因不是“最快”，而是：
- 上限高
- 后续扩展空间大
- 更适合管理员后台、AI 模块、日志模块、副线调度、叙事模块继续增长
- 更适合后续部署到服务器

## 2. 为什么不用一体化先写
当前项目不是一次性 demo，而是会持续扩展的实验平台。

因此当前不走“前后端完全揉在一起”的低门槛方案，而是从一开始就保留：
- 清楚的前后端边界
- 独立数据库
- 后续可单独部署和扩展的能力

## 3. 当前项目结构方向
当前建议采用 monorepo，后续可按下面方向组织：

- `apps/web`：Next.js 前端
- `apps/server`：NestJS 后端
- `packages/shared`：前后端共用类型 / 常量 / 工具（后续再补）
- `packages/config`：共用配置（后续再补）

当前原则：
- 先把骨架搭出来
- 不为了显得高级而乱加层
- 但必须为后续服务器部署和模块扩展留口

## 4. 第一阶段开发必须考虑的部署约束
虽然第一步先本地跑，但写代码时必须默认以后要上服务器。

所以当前要求：
- 配置走环境变量，不写死本地路径
- 数据库连接字符串可切换
- 前后端地址可切换
- 文件 / 材料存储接口后续可替换
- 代码不要假设永远只跑在本机
